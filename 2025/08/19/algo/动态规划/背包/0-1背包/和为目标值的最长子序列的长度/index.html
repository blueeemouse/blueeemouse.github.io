和为目标值的最长子序列的长度

Posted on 2025-08-19 In algo , 动态规划 , 背包 , 0-1背包

原题链接在这

题目描述：

给你一个下标从 0 开始的整数数组 nums 和一个整数 target 。

返回和为 target 的 nums 子序列中，子序列 长度的最大值 。如果不存在和为 target 的子序列，返回 -1 。

子序列 指的是从原数组中删除一些或者不删除任何元素后，剩余元素保持原来的顺序构成的数组。

示例 1：

输入：nums = [1,2,3,4,5], target = 9

输出：3

解释：总共有 3 个子序列的和为 9 ：[4,5] ，[1,3,5] 和 [2,3,4] 。最长的子序列是 [1,3,5] 和 [2,3,4] 。所以答案为 3 。

示例 2：

输入：nums = [4,1,3,2,1,5], target = 7

输出：4

解释：总共有 5 个子序列的和为 7 ：[4,3] ，[4,1,2] ，[4,2,1] ，[1,1,5] 和 [1,3,2,1] 。最长子序列为 [1,3,2,1] 。所以答案为 4 。

示例 3：

输入：nums = [1,1,5,4,5], target = 3

输出：-1

解释：无法得到和为 3 的子序列。

提示：

  • 1 <= nums.length <= 1000

  • 1 <= nums[i] <= 1000

  • 1 <= target <= 1000

    思路：

    这题和之前的题目的主要区别在于，它要的是符合要求的最长子序列的长度，而不是符合要求的子序列的个数。那之前我们构造的record数组一般都是表示合法方案的个数的（并且联想一下递归/递推的思路，确实如此）。但这题要“长度”，如果还用之前的思路，显然是不行的。一个直观的思路是说，我们在维护方案数的同时，也额外维护“合法子序列的最长长度”。那自然的，会想到构造一个三维vector。核心代码如下（抛开初始化之类的细节）：

    解读的重点就是对于record的形状的含义的理解。它本质上就是在以前的二维record的基础上增加一个维度，用于维护“合法子序列的最长长度”，，所以以前二维的record，其ij元record[i][j]表示从下标为0到下标为i的数组元素中，和为j的挑选方案数；现在的三维的record，其ijk元record[i][j][k]含义为：

    record[i][j][0]表示从下标为0到下标为i的数组元素中挑选元素，是否存在和为j的方案；
    record[i][j][1]则表示，如果存在和为j的方案，那么在所有的方案中（也就是在所有的子序列中），子序列的最长长度

    逻辑上这是没问题的，事实上也能通过给的几个简单的测试样例，但最后是通不过这道题的。经过若干次测试，发现大概会卡在920左右的测试样例（总共941个）。然而那些可通过的代码，它们的时间复杂度其实和上面的代码也是一个级别的，都是\(O(n*target)\)级别的，按理说这里的代码也应该能通过才对。但实际无法通过，那应该就是在实操的时候，上面的代码还是有一些细节的时间开销比较大（毕竟时间复杂度衡量的是数据量足够大的时候的时间开销量级，而测试样例可能还没那么大的数据量，所以时间表达式里的常数项依然发挥着作用）
    可能的原因也有很多，比如这里构造了三维的vector，那么初始化的时候就需要花更多时间；又比如在两重的for循环里面，有太多复杂的判断；以及可能现在的代码压根就没有去注意硬件协同，所以缓存局部性之类的没有利用好……总之，上面的代码可以用作启发、思考，但是无法通过题目

    改进其实也就是从“三维”这个问题入手。真的一定要三维吗？不可以用二维吗？其实是可以的。只不过此时二维的元素表达的含义就不太一样了。我们不妨直接维护合法方案（也就是合法子序列）的最长长度；如果不存在合法子序列，我们可以规定其值在某个范围里。这样，我们就只需要构造二维的vector了。最后也发现，二维的时候就能通过了。具体可以看下面的代码

    我们先把所有的元素初始化为INT_MIN，也就是很小的整数（实际是一个绝对值很大的负数，有这个大概感觉即可）

    先看看初始化的部分：record[0][nums[0]] = 1, record[0][0] = 0;，这里很好理解，第一个维度索引都是0，所以表示的是从下标0到下标0的这些元素中进行挑选（其实就是选还是不选第一个元素）；第二个索引为nums[0]，表示从第一个元素中，挑出一个元素，使得子序列和为nums[0]（那显然就是要取第一个元素），此时的子序列长度为1，故该元素赋值为1；而第二个索引为0的时候，就表示从第一个元素中，挑出元素，使得子序列和为0（根据题意，此时只可能是不取第一个元素），所以子序列长度为0，故赋值为0

    再讲一下一些代码优化的细节。上面初始化的时候，总是有vector<vector<int>> record(n, vector<int>(max(target, nums[0])+1, INT_MIN));这一步，主要是这里的取max操作，会导致，当target比较小，但nums[0]比较大的时候（这是有可能的），我们初始化了一个比较大的二维vector；但是下面的二重循环里，我们在第一个维度上最多遍历到n-1，在第二个维度上最多遍历到target，这就导致二维vector里可能会有很多位置我们压根就没用到，无疑是浪费空间了（虽然并不影响时间复杂度，但会影响空间复杂度）。最简单的优化，可以参考下面的代码：
