Ever-Evolving Memory by Blending and Refining the Past

Posted on 2025-09-05 In 论文阅读 , nlp , llm , RAG , memory management

Venue：arxiv

date：2024-04-07

动机：

论文关注的问题依然是长期对话里的长期记忆构建、管理。整体上和[[Keep Me Updated! Memory Management in Long-term Conversations]]研究的问题很像，方法也很像
管理长期记忆的时候，除了记忆矛盾与冗余这个问题（也就是历史记忆里有过时的、矛盾的或者冗余的信息），还有记忆碎片化的问题（这个应该算是论文比较大的创新点了。因为前一个问题研究的还是比较多的。这个问题关注的是，当我们要从多个会话里提取信息、更新记忆的时候，除了要获得一个良好的、更新后的记忆，我们也希望能保留一定的“过程”。就是说，更新的这个过程其实也是可以记录下来的。比如，历史记忆里曾经有“用户生病”这个信息；当前会话表明用户已经康复；那么我们除了把“生病”这个矛盾的消息给剔除掉，“用户曾经生病，现在康复了”这个过程性的信息，其实也可以保留（这点算是[[Keep Me Updated! Memory Management in Long-term Conversations]]没做到的）

简介：

论文的CREEM框架，大体上分三个步骤：

1. 上下文检索（Contextual Search）。这一步其实和naive RAG差不多，会根据当前对话检索相关的历史记忆。不过除此之外，值得注意的是，论文还多了一步，利用llm，根据当前对话，总结出一个query，然后也会用这个query检索一下。也就是这里会检索两次。用llm总结一个query，可以认为利用了llm的总结和理解能力，从而总结得到的query有可能代表了当前对话的更高层次的信息，基于此有可能检索到一些补充性的历史记忆

2. 融合与精炼（Blend & Refine）。这一步算是核心，它完成的正是对记忆的管理。我们的目标是要让记忆能保有新的信息，剔除过时信息和矛盾的信息。在此之上。论文还多了一个小细节（也是相对重要的点），即保留了一定的总结性信息（这个是我自己命的名。在论文中，这个叫做洞察insight。简单说，这个insight分三类，一类是更新型（Updated），此时会生成一个体现了变化的insight。可以参考上面用户生病的那个例子。第二类是冗余型（Redundant），这类是说，如果历史记忆和当前信息重叠了，那么insight就会把二者合并起来，类似于生成一个归纳性的结论。第三类是新增型（New），如果当前信息是完全新的消息，则直接生成新的insight，也就是没有其它的处理）
论文的“融合”（Blend）步骤，就是在生成insight；而“精炼”（Refine）步骤就是剔除无效的记忆（冗余的、过时的）

3. 注入insight的回复生成。这一步就是利用生成的insight、更新后的记忆和当前的对话上下文，完成当前的回答。由于信息更多更充分了，因此回答得更好

insight：

关注到了一个不太有人关注的小细节，即论文中所谓的insight。它包含了一定的归纳性、过程性的信息，是一个比较好的补充

此外，检索的时候，多了一步用llm总结出query并检索。通常，我们只用当前上下文来检索。其实可能需要考虑，一般的dense retrieval它是否有足够的推理能力，以至于我们只用当前上下文来检索就能检索到我们想要的内容（然而应该并不够。否则也不需要有那么多论文提出各种检索的方法，什么多次迭代检索生成啊，反过来先生成再检索啊之类的。而且最近Google deepmind不也有论文揭示了embedding检索的固有缺陷嘛）。这里用llm总结出query，其实就是某种程度上用llm提取一些相对高层次的信息，从而进行一个补充的检索（如果直接用当前上下文检索，就类似于仅局限于字面意义的检索）

除此之外，其实没有太多创新的好像。尽管是24年的paper了，可是和22年的[[Keep Me Updated! Memory Management in Long-term Conversations]]似乎也没太多差别，主要的注意力还是放在如何更新管理memory，使得其包含新的、不冗余的、不矛盾的信息，而手段依然还是定义几个操作，再让模型输出应该采取哪个操作，再更新记忆。真的没有太多新意啊
