kv cache基础

Posted on 2025-09-27 In blog阅读 , llm , efficiency , kv cache

在了解kv cache的过程中，读到了游凯超的一篇回答，感觉真的写的很详细，而且不仅局限于kv cache，包括为何这样设计，也讲清楚了

首先是关于为什么要有kv cache（以及为何只要kv cache，而不是其它的cache）

简单来讲，就是在自回归生成token的过程中，我们发现过往token的一些东西其实是可以复用的。不妨考虑第l层，我们已有n个token（这是包括了prompt以及所有生成的token的总数），且生成了第n+1个token。那之后就需要让这总共n+1个token重新经过总计L层的模块，得到最终的、聚合后的、第n+1个token的embedding，用于生成第n+2个token

而且我们可以发现，在经过L层的过程里，其实前n个token的embedding，在每一层，都是和之前一样的。也就是，我们不需要重新算它们。只需要算第n+1个token在各个层的embedding。因此，一个朴素的想法是，我们保留每一层前面的token的embedding，只计算新的token的embedding即可。这个想法可以被称为x cache，因为它存储的是token在各层的embedding。

那为什么现在流行的并不是x cache，而是kv cache呢？这就要结合到每一层具体的计算方法了。在x cache设定下，每一层我们对第n+1个token的embedding的计算，其实就是一个single-query attention。因为这里我们是要用最后一个token的query，与前n个token的key进行计算，从而得到attn，之后进行聚合（就是对v进行加权求和），得到最后一个token的embedding（当然，如果是多头注意力，可能需要把各个头的输出进行一个拼接）写出公式会更加明显看出复用在哪里：
\[s_{i}=(W_{Q}\cdot x_{n+1}^{l})^{T}(W_{K}\cdot x_{i}^{l})\ \ \ i=1\dots n+1\]\[y_{h}=\sum_{i=1}^{n+1} \frac{e^{s_{i}}}{\sum_{j=1}^{n+1}e^{s_{j}}}(W_{V}\cdot x_{i}^{l})\]\[x_{n+1}^{l}=Concat_{h=1}^{H}(y_{h})\]因为是x cache，所以中间的\(\{ x_{i}^{l}|1\leq i \leq n \}\)都是保存下来了的。但即使这样，我们依然要计算\(W_{Q}\cdot x_{n+1}^{l}\)和\(W_{K}\cdot x_{i}^{l}\)，以及\(W_{V}\cdot x_{i}^{l}\) 后两项都是比较多的，要计算n个
既然这样，为什么我们不把\(W_{K}\cdot x_{i}^{l}\)和\(W_{V}\cdot x_{i}^{l}\)直接保留下来呢？这样就剩去了前面token的相关计算部分（当然，第n+1个token相关的计算是免不了的，毕竟它是刚刚生成的，没有cache的机会）。而保留下来的这两项，其实就是之前token的k和v。所以，出现了kv cache

原回答里还有一个很有意思的点，是，它提出，其实保存哪个，都是看有多方便我们计算的。x cache也是有用的，毕竟省去了一部分的计算，但是还留有一部分计算。而kv cache则是改进了x cache，省去了更多的计算。如果我们把\(s_{i}=(W_{Q}\cdot x_{n+1}^{l})^{T}(W_{K}\cdot x_{i}^{l})\ \ \ i=1\dots n+1\)这个计算过程给改写一下，中间的qk计算attention score的部分，我们写为：\((x_{n+1}^{l})^{T}\cdot W_{Q}^{T}\cdot W_{K}\cdot x_{i}^{l}\)，则把后面三项结合起来，然后存储，也是一种方案（也即\(W_{Q}^{T}\cdot W_{K}\cdot x_{i}^{l}\)这部分存起来）。可以把这种存储方法叫做s cache。当然，这种情况下，我们还是需要把v cache存起来的。否则仅有s cache的话，后面聚合的那一步我们还是要计算\(W_{V}\cdot x_{i}^{l}\)的，这还是很麻烦的。所以严格说可能这种方法叫sv cache更合适
总之，其实cache就是为了复用。所以存哪些东西，就看我们的计算过程中哪些东西是可以复用的。并且通过适当的改写，有可能存的东西会变化

当然，原回答还对比了这三种cache方案的计算开销和存储开销。结论是，一般情况下，这三个方案的计算开销其实都是一个量级的，因此考虑选存储开销最小的那个，也就是kv cache了。这也解释了为什么有kv cache，为什么是kv cache而不是其它cache

这些东西，似乎现在一般都囊括在框架里了。有点细节以及麻烦，可能得读一下vllm之类的框架的源码，或者读一下相关论文了

最后是kv cache的成立条件

这里，原回答讨论的是，kv cache这种加速技术能否用于所有的llm。答案是不行

可以抽象成一个数学问题来看。前面我们提到了，kv cache的提出是因为发现计算过程中有东西可以复用，可以复用的东西分别是之前token的k和v。这个的意思是说，即使我们要计算第n+1，n+2，……个token，前n个token的kv都是不变的。也就是说，如果我们把llm的每一层视为一个函数f，它的输入是若干token，且这个输入是会变长的。以其中一个层的计算为例，\[(x_{1}^{l+1},\cdots,x_{i}^{l+1},\cdots,x_{n}^{l+1})=f(x_{1}^{l},\cdots,x_{i}^{l},\cdots,x_{n}^{l})\]如果后续有更多token来了，假设是m个，则计算过程就变成了\[(\hat{x}_{1}^{l+1},\cdots,\hat{x}_{i}^{l+1},\cdots,\hat{x}_{n}^{l+1},\hat{x}_{n+1}^{l+1},\cdots,\hat{x}_{n+m}^{l+1})=f(\hat{x}_{1}^{l},\cdots,\hat{x}_{i}^{l},\cdots,\hat{x}_{n}^{l},\hat{x}_{n+1}^{l},\cdots,\hat{x}_{n+m}^{l})\]如果此时满足\(\hat{x}_{i}^{l}=x_{i}^{l}\ ,\ \ \ 1\leq i \leq n\)，则kv cache依然适用。因为它表明，新来了token也不影响我之前token的embedding，当然也就不会影响之前token的kv了（推理的时候权重是不变的，因此kv是否变化只看token本身的embedding是否变化了）
这个性质可以被称为因果性，也就是每个天空嗯的输出只依赖于它自己以及它之前的token。因此，那些decoder-only的模型似乎都是满足这一点的。因为它们都是生成模型，采用自回归的形式，且都会采用causal mask。而encoder-only的那些模型，比如BERT类，它们就不满足这个性质。因为它们是双向的注意力，每个token的输出和它之后的token也有关系
现在的llm大多都是decoder-only的结构了，那是否意味着kv cache能适用于所有的llm呢？
也不是。这里我们忽略了对输入的预处理。一般的预处理，除了把token idx映射到embedding，还需要加上position embedding。而问题就出在这里。有一些pe方法，它们在输入增长之后，会整体重新计算一下，因此前面token的embedding也会发生变化。所以，此时前面的token的初始的输入就已经发生了变化，那后面我们自然也不能保证embedding还是相同的。这种情况下，kv cache还是不适用的

所以，kv cache要适用，至少得满足以下条件：

模型结构得具有因果性

pe也具有类似的因果性
