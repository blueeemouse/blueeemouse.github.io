python项目中导入路径的经典报错

Posted on 2025-11-25 Edited on 2025-11-26 In python相关bugs

在部署一个定时爬取arxiv论文的小项目的时候，又一次遇到这个经典的问题了。所以尝试彻底解决一下

问题描述：

在比较大点的python项目里，我们会把一些自己写的文件放到一个文件夹里，然后当作一个库，在其它文件中进行导入。但是，有时会出现，明明这个路径下是有对应文件的，导入的时候却报错ModuleNotFoundError: No module named ‘xxx’。例如，

所以问题在哪里呢？

原因分析：

在于工作路径。python导入模块，会按照顺序搜索相对路径的起点：
- 当前脚本所在目录
- PYTHONPATH 环境变量指定的目录
- Python 安装的标准库目录
- .pth 文件指定的目录
（而且一般只会在当前脚本所在目录下，其它的如果不指定，是不会去搜索的）
搜索到相对路径的起点之后，再拼接上导入的路径，作为最终完整的路径。可以参考这个图：其中搜索路径列表就是我们上面提到的那几个按顺序搜索的路径，而拼接路径后面跟着的模块名，对应我们上面提到的，在文件里具体的怎么导入的，或者说是文件里的导入形式（如果进行了多个搜索，那只要其中一个搜索的路径确实存在，就可以）

如果发现这个完整的路径是不存在的，就会报上面的错。那回到上面的例子，当我们在test路径下时，工作路径是当时所处的test路径，也即d:\daily-arxiv\test，然后导入的路径是src.utils，那拼接得到的路径就是d:\daily-arxiv\test\src\utils，然而这个路径确实不存在。所以会报错
当我们在arxiv-daily路径下，运行python test/test_fetecher.py的时候，工作路径是d:\daily-arxiv，则拼接后的路径是d:\daily-arxiv\src\utils，那这个路径是正确的，所以能导入

另一个实例&tips

同理，需要注意的是，我们导入的文件，它可能需要导入其它地方的路径。此时的工作路径依然是和我们现在使用的路径是一样的，而不是看那个文件所处的路径（也就是，如果我们在文件a下，导入了其它地方的文件b，文件b里面又导入了其它地方的文件c，则b在导入这个c的时候，依靠的还是运行a的时候所处的工作路径A，而不是依靠文件b所处的路径B）

举个例子，在上面的路径下，我们运行test路径下的test_fetcher.py文件。它导入了src里的utils文件。而utils文件导入了config里的配置文件。事先声明，src里的utils文件，它默认是通过config/config.yaml路径导入配置文件的。config目录下确实是有这个config.yaml文件的
然而如果我们是在test路径下运行python test_fetcher.py，不改utils里的配置，就会报错。还是可以如上分析：因为我们现在在test路径下，那拼接完成后的路径是d:\daily-arxiv\test\config\config.yaml，当然会报错。所以如果要在test路径下运行，得改改utils里导入的方式，譬如改成../config/config.yaml。这样，拼接得到的路径就是d:\daily-arxiv\test\..\config\config.yaml。注意，一个.代表往上走一级目录，所以上面的路径往上走两级，其实就走到了daily-arxiv路径下了，最终等效于路径d:\daily-arxiv\config\config.yaml。这就正常了。所以能运行不报错
类似的，可以想到，如果我们是在daily-arxiv路径下运行python test/test_fetcher.py，就不需要改成../config/config.yaml，保持config/config.yaml就可以了（不如说此时不改才会报错）

道理上就是这样的。实操上想避免，最根本的还是从写项目开始就把这些细节统一好，然后在写readme的时候也讲清楚（不过事实是很多时候我们是在运行别人的项目的时候遇到这个问题的，并且项目里也没有写清楚是怎么回事……）
