REPOGRAPH：ENHANCING AI SOFTWARE ENGINEERING WITH REPOSITORY-LEVEL CODE GRAPH

Posted on 2025-07-10 Edited on 2025-07-11 In 论文阅读 , nlp , llm , llm4code

date：2025-03-18

动机：

实际软件开发里涉及的代码的体量较大，是repo-level的，而非一般的function-level或file-level。虽然现在有各种coding llm，在file-level及以下的各种任务（如补全、debug等）都做得比较好，但迁移到repo-level任务上表现欠佳
原因在于repo-level的代码文件之间有复杂的依赖关系，因此整个repo内部是有结构的，要在repo-level的任务上表现好，就需要能掌握这种结构。现在的各种方法对这个结构没有很好地把握住
因此考虑提出用graph结构来更好解读、利用这种repo里的结构

insight：

用graph结构来定位和理解。把代码之间的关系抽象成graph，其实是稍微会好一点的，比起直接把所有的文件一起当作输入

contribution：

提出了一种plug-in module，可以适配现有的方法，能增进对代码仓库的整体理解以及定位

现有方法不足：

文章提到三类方法，一类是RAG类，直接搜索和当前文件相关的其它文件（但这个本质上只是在搜索相似的代码片段，可能会遗漏其它方面的相关的文件）
另一类Agentless的，它提出对每个文件都提炼得到一个sketch，然后把所有文件的sketch都作为prompt的一部分交给llm让它进行解析（这种方法，或许可以理解为，它是在让llm自己去理解文件之间的关系，并没有显示地提炼；整体上还是把仓库视为“放了很多个文件的集合”）
最后是把rl用在llm里，即让llm自行和仓库进行交互，从而理解仓库。这里文章提到问题在于，因为没有对仓库的结构的理解，llm最后和仓库的交互，可能收敛到和某一个文件的这种局部交互，导致对结构的理解依然不好（？）

method：

论文主要是提出REPOGRAPH，是一个即插即用的模块。它的输入是一个repo，也就是一个仓库，包含了各个文件夹以及文件；输出则是一个structured graph，结构化的图，节点是code line（这个得理解一下，难道真的是一行代码？那也太抽象了），而边则是代码之间的关系
用这个图结构表示，则当我们拿到一个issue的时候，可以根据graph追溯到issue的根源节点，然后把相关的代码都拿给模型，用于解决问题
那么下面的关键就是如何根据repo，构建出相应的graph；以及如何应用构建出来的graph

1. 构建graph分三步：

Step1. Code line parsing

简单说，它是对每个文件都抽出重要的代码行，有待用于后续的处理
流程上，它会遍历仓库的文件（仅限特定后缀，这样可以避免对所有文件都进行遍历，减小工作量），对每个文件都抽出抽象语法树Abstract Syntax Tree（AST），用来表征文件里的逻辑关系；
之后针对AST抽象出的各种元素，我们不仅要知道它们的定义，还需要知道它们在哪里被调用或者引用了（这样才能更加清楚代码之间的依赖关系）
最后，我们要关注那些AST元素的定义的代码行（比如，类的定义的代码行，函数定义的代码行），以及这些元素被引用的地方的代码行（例如，类在哪里被实例化了）（此处论文仅关注元素里的函数和类，其余的不关注。因为这两个元素是比较重要的，保留这些相关的就能掌握大部分信息，同时能减少计算开销，便于后续的处理）

Step2. Project-dependent relation filtering

概括地讲，这一步我们是要把上面提取出的重要代码行进一步进行筛选，筛出那些和我们的需求（比如issue）相关的代码行
大体思路是，我们上面是根据元素的定义和引用来筛出代码行的，但这里有一些其实是肯定没必要的。比如，有一行代码调用了python自带库的函数，例如len()函数，那它确实有调用关系，调用了一个函数，但这个函数很显然不重要，所以需要筛掉；又比如，调用了第三方库的函数，这里因为调用的不是我们自己写的代码，所以也忽略掉。那么总结一下，其实就是说，project-independent relationships有两类，一类是global relation（指的就是调用python自带库、内置函数、内置类等），另一类是local relation（指的是调用第三方库的东西，而不是调用我们文件里自己写的东西）（因为这种关系，并没有反应我们的repo内部的依赖关系，属于是干扰项了）

Step3. Graph organization

这一步就是要把上面我们进一步筛选过的代码行作为节点，根据各个代码行之间的依赖关系，构建得到graph
其实大体上已经完成了，剩下就是对边和节点的一些细化定义
我们定义每个节点代表一个code line，它会包含一些属性attributes，来表征这个代码行的一些信息（例如，它是哪个文件里的，是在第几行，所在的目录是……）；节点分两类，一类是definition（def），一类是reference（ref）。前者表示某个元素在这个代码行里得到了初始化，后者表示某个元素在这个代码行里被引用或者调用了
边则是由两类：\(E_{invoke}\)和\(E_{contain}\)
前者表示两个节点之间的引用关系，一般会有这样一个三元组\((V_{1},E_{invoke},V_{2})\)来表示这种关系，即节点\(V_{1}\)在节点\(V_{2}\)处得到了引用；后者表示两个节点之间的包含关系，三元组\((V_{1},E_{contain},V_{2})\)表示节点\(V_{1}\)包含了节点\(V_{2}\)

2. 应用graph（Utility）

论文提到了这个repograph方法是即插即用的，所以应用部分其实主要探讨的是如何融入到现有的框架里。文章主要讨论两种框架，一种是procedural framework，另一种是agent framework

融入到procedural framework：

procedural framework，它大致上分两步，即定位-编辑。首先要做的是定位出问题的代码，之后就是看看应该怎么修改。定位和修改的输出建议都是由llm给出的，prompt就是相应的代码段。而repograph融入进来的方法，则是在这两步操作前，把我们提取出来的ego-graph（子图）加到prompt里
比如，我们要解决一个issue，issue里的关键词可能是seperability matrix，那么我们就根据这个关键词，搜索以这个关键词为核心的k-hop子图，然后把子图的内容转换成自然语言（此即文中的flatten ego-graph操作，graph是一种结构化的数据，不适合直接作为llm的输入，需要转成更常用的自然语言）。这就是在定位阶段要用到的子图信息，我们把它和现有方法得到的输入一起作为prompt给llm即可。至于在编辑阶段，也是类似的，我们找到了要修改的代码之后，也把相应的子图作为prompt的一部分送到llm里

融入到agent framework：

agent framework，它需要agent和代码仓库进行交互。而把repograph融入进去的方法，就是在agent的动作空间里，多加上一种动作：search_repograph（也就是上面提到的，根据一个item，搜索得到k-hop的子图），至于什么时候采取这个动作，就不像上面那样有明确的规定，而是由agent自己决定了；而如果采取了search_repograph，那么得到的子图也就是observation的一部分

analysis

论文还分析了一下repograph的效果，包括：分析repograph对问题代码定位精度的提升；repograph的其它的融入到现有框架的方法；以及repograph的error bound

对问题代码定位精度的提升：

结果上看，不管是应用到哪个框架上，不管是哪个粒度上（file-level，function-level，line-level），定位精度都有所提升。只不过确实是越细的粒度上，提升越小。这是符合直观预期的，毕竟粒度越小，定位难度是越大的（以及，看表格可以看到，现在的方法在line-level和function-level上的精度其实感觉都不太够。这应该确实极大限制了最终的效果。不过话又说回来，有一个问题是，它这里相当于有一个标准答案吧，就是认为改哪些代码是对的；但其实，有没有可能出现这种情况：我不改标准答案里的那些代码，而是去修改其它地方的代码，最终也可以解决问题？如果llm/agent真的就是这样做了，那确实解决了问题，但也确实会被判定为没定位精准。这样看，似乎定位精度并不能完全代表最终的效果。事实上，agent framework的定位精度是高于procedural framework的，然而最终效果上是后者更优。论文给出的解释，大体上是说，agent framework它是基于trial-and-error方法的，初期的一些error可能会传播导致影响到后面。这个姑且可以当成是一种部分的解释吧）

其它的融入到现有框架的方法：

上面提到，获取子图的时候，是得到一个k-hop子图。这个k理论上来说当然是越大越好，但越大，越有可能收集到无关的信息。尽管理论上llm能力够强的话，它应该可以辨别、剔除掉那些无关的，仅把有关的保留下来，但实操上，k过大很可能因为引入了过多的无关信息，反而降低最终的性能。另一方面，k越大，则子图越大，且很有可能是指数级增大的规模，所以计算开销也会相应增大，这也限制了我们不可能用太大的k。论文这部分主要讨论了k为1和2的情况
另外还有操作余地的地方，就是如何提取出子图的信息。直接展开，转成自然语言，当然是可以的，这也是正文里提到的。不过这样可能会占很多token，进而很可能超出llm的上下文限制，反而可能影响效果。这部分里，论文讨论了另一种融入的方法，就是先用一个llm，对子图进行提取和化简，之后再做其它处理。这样先做一步化简的好处是，最终得到的token会更少，更不容易超出上下文限制。然而，如果本身子图的信息不够丰富，例如，k为1，此时再让llm进行summarization，可能会适得其反，因为这种情况下再化简，可能漏掉了信息

实验结果表明，确实如此。当k为2的时候，因为子图很大，所以直接展开token太多了，超了上下文限制，反而加上子图信息的时候效果降低了；而如果用了llm进行summarization，则效果提升
当k为1的时候，子图没那么大，则直接展开效果更好（事实上是这里的四种设定里最好的），因为没丢信息，也不至于超出上下文限制；此时进行summarization之后则效果降低了

迁移能力测试

这部分其实是更多是想看一下，repograph在其它任务里能不能提升效果吧。正文里的实验，更多是测试repograph在解决repo-level的issue的时候，能不能通过提升模型对repo整体的感知，从而改进issue解决的效果；这里则是测试repograph应用到repo-level的代码补全的时候，能不能提高对repo的整体理解，进而提高生成质量
实验结果表明是可以的

error analysis

在这一部分，论文研究了加入repograph前后，模型犯错的个数以及各种错误累别的个数，想看一下repograph对错误例子的分布是否有影响（这个思路还是比较独特吧）

首先是定义了三类错误，第一类是因定位错误导致的错误，第二类是因为和更大范围的上下文产生冲突（比如修改当前文件里的、定位正确的代码，但是和其它文件里的代码产生了冲突）而产生的错误，第三类是在修改编辑代码的过程中引出的新的问题

对于两类框架，加上repograph之后，各类错误都减少了，并且用了repograph之后，如果对比每个框架加上repograph前后的解决成功的案例，加上repograph才成功解决、而没加上repograph就没解决的案例是相对比较多的；没加上repograph才解决成功、加上repograph就失败的案例是相对比较少的。这可以从侧面印证repograph的能力（以及，加入repograph之后，确实对localization error效果显著）

    1. 7.1. 论文还分析了一下repograph的效果，包括：分析repograph对问题代码定位精度的提升；repograph的其它的融入到现有框架的方法；以及repograph的error bound
    2. 7.2. 对问题代码定位精度的提升：
      1. 7.2.1. 结果上看，不管是应用到哪个框架上，不管是哪个粒度上（file-level，function-level，line-level），定位精度都有所提升。只不过确实是越细的粒度上，提升越小。这是符合直观预期的，毕竟粒度越小，定位难度是越大的（以及，看表格可以看到，现在的方法在line-level和function-level上的精度其实感觉都不太够。这应该确实极大限制了最终的效果。不过话又说回来，有一个问题是，它这里相当于有一个标准答案吧，就是认为改哪些代码是对的；但其实，有没有可能出现这种情况：我不改标准答案里的那些代码，而是去修改其它地方的代码，最终也可以解决问题？如果llm/agent真的就是这样做了，那确实解决了问题，但也确实会被判定为没定位精准。这样看，似乎定位精度并不能完全代表最终的效果。事实上，agent framework的定位精度是高于procedural framework的，然而最终效果上是后者更优。论文给出的解释，大体上是说，agent framework它是基于trial-and-error方法的，初期的一些error可能会传播导致影响到后面。这个姑且可以当成是一种部分的解释吧）
    3. 7.3. 其它的融入到现有框架的方法：
      1. 7.3.1. 上面提到，获取子图的时候，是得到一个k-hop子图。这个k理论上来说当然是越大越好，但越大，越有可能收集到无关的信息。尽管理论上llm能力够强的话，它应该可以辨别、剔除掉那些无关的，仅把有关的保留下来，但实操上，k过大很可能因为引入了过多的无关信息，反而降低最终的性能。另一方面，k越大，则子图越大，且很有可能是指数级增大的规模，所以计算开销也会相应增大，这也限制了我们不可能用太大的k。论文这部分主要讨论了k为1和2的情况另外还有操作余地的地方，就是如何提取出子图的信息。直接展开，转成自然语言，当然是可以的，这也是正文里提到的。不过这样可能会占很多token，进而很可能超出llm的上下文限制，反而可能影响效果。这部分里，论文讨论了另一种融入的方法，就是先用一个llm，对子图进行提取和化简，之后再做其它处理。这样先做一步化简的好处是，最终得到的token会更少，更不容易超出上下文限制。然而，如果本身子图的信息不够丰富，例如，k为1，此时再让llm进行summarization，可能会适得其反，因为这种情况下再化简，可能漏掉了信息
      2. 7.3.2. 实验结果表明，确实如此。当k为2的时候，因为子图很大，所以直接展开token太多了，超了上下文限制，反而加上子图信息的时候效果降低了；而如果用了llm进行summarization，则效果提升当k为1的时候，子图没那么大，则直接展开效果更好（事实上是这里的四种设定里最好的），因为没丢信息，也不至于超出上下文限制；此时进行summarization之后则效果降低了
    4. 7.4. 迁移能力测试
      1. 7.4.1. 这部分其实是更多是想看一下，repograph在其它任务里能不能提升效果吧。正文里的实验，更多是测试repograph在解决repo-level的issue的时候，能不能通过提升模型对repo整体的感知，从而改进issue解决的效果；这里则是测试repograph应用到repo-level的代码补全的时候，能不能提高对repo的整体理解，进而提高生成质量实验结果表明是可以的
    5. 7.5. error analysis
      1. 7.5.1. 在这一部分，论文研究了加入repograph前后，模型犯错的个数以及各种错误累别的个数，想看一下repograph对错误例子的分布是否有影响（这个思路还是比较独特吧）
      2. 7.5.2. 首先是定义了三类错误，第一类是因定位错误导致的错误，第二类是因为和更大范围的上下文产生冲突（比如修改当前文件里的、定位正确的代码，但是和其它文件里的代码产生了冲突）而产生的错误，第三类是在修改编辑代码的过程中引出的新的问题
      3. 7.5.3. 对于两类框架，加上repograph之后，各类错误都减少了，并且用了repograph之后，如果对比每个框架加上repograph前后的解决成功的案例，加上repograph才成功解决、而没加上repograph就没解决的案例是相对比较多的；没加上repograph才解决成功、加上repograph就失败的案例是相对比较少的。这可以从侧面印证repograph的能力（以及，加入repograph之后，确实对localization error效果显著）
