Code Graph Model (CGM)：A Graph-Integrated Large Language Model for Repository-Level Software Engineering Tasks

Posted on 2025-07-15 Edited on 2025-07-16 In 论文阅读 , nlp , llm , llm4code

Venue：arxiv

date：2025-06-23

动机：

感觉动机这一块其实挺常见的。llm发展很快，将其应用到单文件的各种代码任务都表现得很好，但是到了更加真实复杂的SE task上，例如repo-level的任务上，它的表现就不够好了

而现有的方法，表现更好的是利用闭源模型的基于agent的方法，但这类方法，因为依赖于闭源模型，所以难以接触到模型权重，因而难以进行改进；同时基于agent的方法，通常推理步骤会很长，因此中间的错误可能会不断积累导致影响决策
所以，为了避免这些问题，我们会想用开源模型，并且避免用agent的方法。但现在基于开源模型的方法，性能还是比上面提到的那类要差一截，还不太能处理repo-level task，因此是有极大的改进空间的
而论文尝试做出的改进，则是想让模型对repo的理解更上一层楼，具体来说是让模型更加了解repo里的代码文件本身的含义，以及相互之间的关系

insight：

怎么乍一看，没什么新奇的呢……也是想提高模型对repo的理解（语义以及结构），也是用graph结构，也是让llm利用graph，顶多是细节上不太一样，框架没怎么变化吧（和RepoGraph对比一下吧……）

contribution：

提出了一个新的架构，大概是在开源的llm上加一些新的模块，用来提高llm对repo的理解，然后可以一起进行微调，从而适应SE tasks，且性能不错（最优的结果还是加上了Graph RAG。但话说，这个设计的Graph RAG是真的出于锦上添花，还是说，本来仅仅用新架构，无法work，或者说性能不够好，所以特意设计了一个Graph RAG来make it work？可能得看一下实验表格里有没有单独用新架构而没用Graph RAG的结果了……）

配套的，还设计了一个Graph RAG的框架，可以进行辅助

细节问题：

1. 基于agent的方法与不基于agent的方法

论文中多次提到，基于agent的方法，因为是多步决策，需要与repo进行多轮的交互，中间可能会有一些小错误，而这些错误的累积可能影响最后的结果。这里简单说明一下二者的大致流程

基于agent的方法：

它会要求与repo多次进行交互，然后依据观测结果（如测试的结果）等来调整决策，逐步推进任务（或许可以类比这个例子：agent先尝试改改这个文件里的某个部分，然后测试一下，发现效果不好，没有新通过的例子，就复原，接着改下一个文件……）

不基于agent的方法：

这类方法，它更加注重对repo的整体理解。一般在接收输入的时候，就会用各种方法去尝试理解repo里代码文件之间的相互关系（比如用graph来表征关系，尝试用各种花里胡哨的方法，在不同粒度建图）
对整体有“充分的观察”之后，再作出决策

简单的想法：

其实感觉上，基于agent的方法，它的核心在于trial-error，和一般的agent/rl没什么区别，只不过是应用到了llm4code这个领域了而已；而不用agent的方法，它注重对repo整体的理解（不像基于agent的方法，是通过不断的试错来逐步获得对repo的理解的），有了整体的理解之后再下手。这样很容易想到，为何不把二者结合起来？也没什么矛盾的地方吧？一开始先用graph等手段，得到对repo整体的理解，然后再用agent来不断尝试。因为有了对repo更全面的理解，所以尝试的时候会更加准确，正确率也会更加低；甚至也可以考虑二者交替着来，比如先用一段时间不基于agent的方法，再用一下基于agent的方法，好像也不是不行。而之所以考虑这样做，是因为这样有望把两种方法的优点都结合起来吧？

2. 节点的属性到底是什么？看起来，似乎节点的属性就是文本，因为它会输入到一个预训练的Encoder（根据论文，是CodeT5+）。如果是这样的话，它是如何存储的？

3. graph-aware attention mask是怎么实现的？得看一下GitHub仓库了

method

因为是用graph的方法，整体还是两步：建图，用图

1. Code Graph Construction

大体上讲，最后我们建立的图，它既有上下级的关系（例如继承），也有平级的关系（例如xx引用xx）。它建图的流程，是先根据AST（抽象语法树），构建出上下级关系，得到一个自上而下的树；之后再用lightweight semantic analysis来分析实体之间的引用、调用等关系，最终得到一个graph

先构建的上下级关系，它涉及多个层次的实体，大到repo（具体来说，一开始构建的树的根节点就是repo，它表征了整个仓库本身），最小到func（也就是函数，而不是像RepoGraph那样，最小到行代码），中间有pkg、file、class等级别的实体

2. Code Graph Models（CGM）

整个的算法，涉及三个部分。一是对前面得到的graph应该如何处理，这里主要是模型架构上的创新；
二是训练的策略；
三是附加的Graph RAG的框架

model architecture：

大体上，这里是利用输入的图数据，增进llm对仓库的理解（语义和结构两方面），分两部分，先是semantic integration，再是structural integration。前者把输入的graph，最终转换到llm的embedding空间，变成若干的节点token（这个节点token会和正常输入的文本token拼到一起，成为最终的token seq）；后者则在处理输入token seq的时候，设计专门的graph-aware attention mask，让节点node之间的交互，仅局限于节点和它的邻居对应的token进行交互

semantic integration：

输入的graph，它的每个节点的属性都是若干的文本（对这个节点对应的实体的描述），首先会经过一个tokenizer，把文本变成token seq。这一步还有个关键的地方，就是chunk的划分。如果一个节点的属性对应的token seq太长了（具体表现为超过512个token），则需要把它划分为若干个chunk，每个chunk最多512个token。并且，每个chunk我们都会单独复制一个节点，此时这些复制的节点，它们可以被视为一个整体：在和其它节点的关系上，它们完全一致，邻居关系和原来的节点保持一致；而在“对内“上，这几个chunk对应的复制出来的节点，它们必须是相互连接的（因为它们本来是同一个节点里的，这个全连接的关系相当于为了让它们保持”联系“）
然后是由text encoder来编码（论文选用的是预训练的CodeT5+），这里会把一个节点的对应token seq压缩到一个向量（对应后续的llm embedding空间里的一个token）（这个操作可以减少上下文的占用，近似提高了512倍的context length）；
之后经过一个adapter，把上述得到的向量转换到llm的embedding空间（所谓adapter就是一个简单MLP）。至此，每个节点都对应了一个token，且是在llm embedding空间的token，可以和文本输入的token一起被处理

structural integration：

到这里，我们已经有了节点对应的tokens以及文本对应的tokens，然后会把它们拼起来得到新的token seq。而为了保证节点对应的token还能呼应上原来graph的结构，我们希望，对每个节点token而言，它只会和原来与它邻接的节点的token进行注意力的交互。这里就需要用专门的attention mask来实现（即graph-aware attention mask），可以由graph的邻接矩阵得到
