RepoCoder：Repository-Level Code Completion Through Iterative Retrieval and Generation

Posted on 2025-07-13 Edited on 2025-07-15 In 论文阅读 , nlp , llm , llm4code

Venue：EMNLP 23

date：2023-10-20

动机：

现有的代码补全工具，通常仅能利用上单个文件里的上下文信息进行补全。但如果要进行repo-level的补全，那就不能仅仅依靠单个文件的信息，而应该对整个仓库的上下文信息有所选择地利用（因为仓库里的文件相互之间是有相较于单个文件更为复杂的依赖关系的）。因此需要专门提出repo-level code completion这个任务进行研究

现有的研究这个任务的方法，主要是两类，
一类是依赖静态代码分析与启发式规则的方法，另一类是基于有标签数据，对语言模型进行微调的方法

第一类，它能可靠地解析上下文，但是补全能力首先，无法生成不同长度的内容，且不支持再文件任意位置进行补全（说到底，可以认为是rule-based的通病）

第二类，主要问题在于泛化性不够好，对于没见过的仓库，要进行补全，效果还是比较差的

insight：

但凡是repo-level的任务，不管是代码理解还是代码生成，抑或是其它的，应该都要求对repo有一个整体的理解，以及在处理片段代码的时候能利用好、检索到正确的上下文。这篇论文最核心的贡献可以说就是在解决这个问题：如何更好地得到正确的、合理的上下文。它的做法是改进RAG，变成多轮的RAG，以期提高检索质量（联想一下之前被的REPOGRAPH，它是通过把仓库的各文件之间的关系抽象为一个graph，用处理graph的方法来处理文件之间的关系，并由此得到正确的上下文。二者似乎是可以融合的。不过这也挺显然的吧，毕竟人家REPOGRAPH设计之初就是一个plug-in的模式，当然可以融合。但更重要的可能是，二者的思路不同，一个是利用外部的模型进行辅助，一个是把数据的形式进行转变。两条思路是并行的，且可以合并的吧）

另外，这个由单轮的RAG变成多轮RAG的迭代的方法，也是一个比较通用的方法，或者说技巧？虽说也有可能因为检索到的结果不够好，反而影响了补全效果，然后下一次迭代时，搜索得到的结果更差，形成负循环，但这也是有改进余地的吧（但话又说回来，看论文2.2 Code Retrieval部分，它也专门提到了，基于之前的补全结果来检索，效果就是能有所提升，即使说补全结果不那么准确。虽然不清楚这个结论的依据是什么……）

还有，用滑动窗口把一个大的内容，分割成若干个小的内容，似乎也是一个实践上常用的技巧

contribution：

提出了一套新的框架，RepoCoder，是一种迭代地使用RAG的框架，可用于repo-level code completion task

提出了一个新的benchmark，RepoEval，可以衡量模型在repo-level code completion方面的能力。且包含了多种粒度

对RepoCoder进行了严格的分析，便于后续的改进

细节问题（阅读论文时遇到的可能的问题）：

1. 论文里提到的三个granularity的具体含义：

line completion（行级补全），即补全代码文件中未完成的单行代码

API invocation completion（API 调用补全）：补全对仓库内自定义 API 的调用语句（如方法调用、函数调用）

function body completion（函数体补全）：补全函数内部的完整实现代码（包含多行逻辑）

2. 补全代码的llm以及检索模型，在整个过程里都是冻结的

3. 常用的评价代码补全的指标，有基于相似性的指标，例如Exact Match（EM）、Edit Similarity（ES）。很显然，这个评价是有失偏颇的，因为我们不能保证原来的代码实现就一定是最优的。即使补全的代码和原代码越相似，确实能在某种程度上表明正确性，但这并不唯一（举个例子，在一个函数的实现里，如果我们仅仅把一个常数给改掉，其它地方不变，那用EM或ES评价，一定都会认为很高质量。但实际很可能算出来的答案永远是错的）。类似的，补全的代码和原代码越不相似，也不能因此就表明它不正确（还是以函数体补全为例，我们的实现大可以和原代码的实现形式上差别很大，但只要逻辑确保是正确的，就没问题。但是如果用EM/ES来评价，可能就会得出补全结果很差的结论，显然有些问题）

4. unit tests 定义：单元测试是用于验证单个函数或方法功能正确性的代码片段，通常包含输入、预期输出和断言语句。（它更加注重逻辑的正确性，而非形式的正确性）

  • 示例：若仓库中有函数def add(a, b): return a + b，对应的 unit test 可能是assert add(2, 3) == 5，用于验证add函数是否正确返回两数之和。

    method

    1. 对RepoCoder框架的大体理解：

    它的大致流程如下：

    进行多轮迭代。其中，每一轮迭代，我们都会先根据待补全的代码，在repo里检索相关的代码片段，加入到上下文里，然后交给llm进行生成补全；
    之后，我们把补全上去的代码，以及之前的待补全代码，拼到一起，再拿去检索repo里相关的代码片段，并仿照上面，把检索得到的新的代码加入到上下文里用于补全

    如果用一些记号进行形式化表达，那么，它的迭代过程大概是这样的：先执行\(C_{ret}^{i}=R(C_{repo},X,\hat{Y}^{i-1})\)；
    再执行\(\hat{Y}^{i}=M(C_{ret}^{i},X)\)
    这就是一轮迭代。其中，记号含义如下：

    \(C_{ret}^{i}\)表示第i个迭代过程里，检索得到的结果；R表示一个检索的模型，\(C_{repo}\)表示repo里的代码（类似于一个数据库），X表示待补全的代码，\(\hat{Y}^{i-1}\)表示上一轮llm补全的结果，M表示llm（用于补全）

    所以，之所以说RepoCoder是一个迭代的框架，是因为我们会重复“检索-生成”这套操作。每一次生成完，我们手头上有的代码更多了，再去检索repo里的相关代码，得到的结果会更加相关、更加可靠，因此新补全的代码质量也更高，而进一步又会帮助优化下一次的检索效果（由此我们似乎可以期望形成一个正循环）

    2. 下面详细讲一下迭代里的两个主要操作：检索和生成

    Code Retrieval：

    当iteration大于1的时候，也就是已经有补全出来的代码了，那么我们的检索就会把补全的代码也加进来（正如上所述）；而如果是初始的iteration，也就是还没有补全出来的结果，则我们只用待补全的代码作检索的query

    检索的时候，retriever不是直接拿着query就去对着仓库里的文件一个个检索。我们需要先做一个预处理，也就是把repo里的代码文件划分成若干个小窗口，每个窗口都是固定行数的代码。构造方法则是滑动窗口（所以这里涉及到两个参数，window size \(S_{w}\)以及sliding size \(S_{s}\)）

    另外，query也会被处理成一个window size。所以，当我们需要把上一轮的补全结果加入到检索的query的时候，query是由最后\(S_{w}-S_{s}\)行的原待补全代码，以及前\(S_{s}\)行的上一轮补全代码组成的

    这里论文提出要把上一轮补全的代码加入到query，其实也是有来历的。这里主要涉及到RAG的生成部分。假设我们现在有原待补全代码片段a，然后检索得到了相似/相关代码片段b，如果我们把b往后移动若干行，得到b'，那b'就可以视为对b的补全。通过参考b'是如何由b进行补全得到的，我们可以迁移到对a的补全。但这里有个问题就是，应该如何往后移动呢？移动的话是全体都往后移动固定的行数吗？行数怎么规定呢？就是说，死板地往后移动，有些情况下效果未必好。为了改进这一点，我们就不通过往后移动检索结果来借鉴了，而是用上一轮的补全结果来借鉴。因此会有“把上一轮的补全结果加入到query”的操作

    Code Generation

    这一部分没有什么好说的，就是把检索的结果和待补全的代码一起当作prompt，拿给llm，让它进行补全。只不过检索结果和待补全代码，我们会用一个固定的模板来把它们组合起来

    benchmark construction

    本文的贡献，除了提出算法层面的新的框架，也包括提出了一个新的benchmark，RepoEval，用于评价repo-level的code completion的质量

    具体构造：

    首先，从GitHub上收集高质量的仓库，并且要是比较新的。补全的粒度有三种，也就是上面提到过的line、API invocation、function body completion。在评价指标方面，评价function body completion的时候，除了基于相似性的指标（EM、ES），还会进行unit test（因为函数体补全，重要的还是它的功能）
    每一条数据，都会有一些“属性”，包括所属的仓库、文件路径、所在行数以及实际的补全代码（也就是标签了）

    之后，需要从收集到的GitHub仓库里进行筛选，要满足若干条件：opensource license, created after January 1, 20223, nonfork original repositories, over 100 stars, over 80% of files written in Python, and explicit unit tests.
    然后，为了避免bias，会从筛选出的GitHub仓库里随机选出若干个，接下来就是相应构造三个粒度的仓库：

    Line Completion：从满足标准的仓库里，随机挑出8个，每个仓库选出200行作为samples（ensuring the lines are non-repetitive, not code comments, each line contains at least 5 tokens.）因此，这里会得到共计1600条数据

    API Invocation Completion：这里我们要选出api的调用，用于补全。还是沿用上面选出的8个仓库，每个仓库选出200条数据。且这里我们关注的是in-repository api，也就是那些仓库里自己实现的api，而不是python自带的，或者第三方库里的。因此，相关的数据更少（因为出了这个仓库，很可能就没有了），难度会更大。
    具体实现上，先通过静态的解析，定位出所有的in-repository api invocation，再从中随机选出200个，作为当前仓库里挑出的数据

    Function Body Completion：此处，因为要补全一个完整的函数，我们评价时需要进行unit test（最后评价的时候，就是计算补全的函数里的通过率，即Pass Rate，PR）。但如果是一个规模很大的仓库，它的unit test很可能量级也很大，依赖很复杂，、实现上的麻烦。因此，构建这个粒度的数据集，我们不再直接沿用上面选出的8个仓库（因为不能保证里面没有很大的仓库），而是专门选出一组规模较小、易于实现的仓库。然后，定位到那些有unit test的函数的具体实现，并选出其中规模为3-30行的，作为测试数据。一共得到了373条数据

        1. 7.2.2.1. 这一部分没有什么好说的，就是把检索的结果和待补全的代码一起当作prompt，拿给llm，让它进行补全。只不过检索结果和待补全代码，我们会用一个固定的模板来把它们组合起来
  • 8. benchmark construction
    1. 8.1. 本文的贡献，除了提出算法层面的新的框架，也包括提出了一个新的benchmark，RepoEval，用于评价repo-level的code completion的质量
    2. 8.2. 具体构造：
      1. 8.2.1. 首先，从GitHub上收集高质量的仓库，并且要是比较新的。补全的粒度有三种，也就是上面提到过的line、API invocation、function body completion。在评价指标方面，评价function body completion的时候，除了基于相似性的指标（EM、ES），还会进行unit test（因为函数体补全，重要的还是它的功能）每一条数据，都会有一些“属性”，包括所属的仓库、文件路径、所在行数以及实际的补全代码（也就是标签了）
      2. 8.2.2. 之后，需要从收集到的GitHub仓库里进行筛选，要满足若干条件：opensource license, created after January 1, 20223, nonfork original repositories, over 100 stars, over 80% of files written in Python, and explicit unit tests.然后，为了避免bias，会从筛选出的GitHub仓库里随机选出若干个，接下来就是相应构造三个粒度的仓库：
        1. 8.2.2.1. Line Completion：从满足标准的仓库里，随机挑出8个，每个仓库选出200行作为samples（ensuring the lines are non-repetitive, not code comments, each line contains at least 5 tokens.）因此，这里会得到共计1600条数据
        2. 8.2.2.2. API Invocation Completion：这里我们要选出api的调用，用于补全。还是沿用上面选出的8个仓库，每个仓库选出200条数据。且这里我们关注的是in-repository api，也就是那些仓库里自己实现的api，而不是python自带的，或者第三方库里的。因此，相关的数据更少（因为出了这个仓库，很可能就没有了），难度会更大。具体实现上，先通过静态的解析，定位出所有的in-repository api invocation，再从中随机选出200个，作为当前仓库里挑出的数据
        3. 8.2.2.3. Function Body Completion：此处，因为要补全一个完整的函数，我们评价时需要进行unit test（最后评价的时候，就是计算补全的函数里的通过率，即Pass Rate，PR）。但如果是一个规模很大的仓库，它的unit test很可能量级也很大，依赖很复杂，、实现上的麻烦。因此，构建这个粒度的数据集，我们不再直接沿用上面选出的8个仓库（因为不能保证里面没有很大的仓库），而是专门选出一组规模较小、易于实现的仓库。然后，定位到那些有unit test的函数的具体实现，并选出其中规模为3-30行的，作为测试数据。一共得到了373条数据
