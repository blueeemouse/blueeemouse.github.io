长度最小的子数组

Posted on 2025-06-05 In algo , 滑动窗口 , 不定长滑动窗口

原题链接在这

题目描述：

给定一个含有 n 个正整数的数组和一个正整数 target 。

找出该数组中满足其总和大于等于 target 的长度最小的 子数组 [numsl, numsl+1, ..., numsr-1, numsr] ，并返回其长度。如果不存在符合条件的子数组，返回 0 。

示例 1：

输入：target = 7, nums = [2,3,1,2,4,3]

输出：2

解释：子数组 [4,3] 是该条件下的长度最小的子数组。

示例 2：

输入：target = 4, nums = [1,4,4]

输出：1

示例 3：

输入：target = 11, nums = [1,1,1,1,1,1,1,1]

输出：0

提示：

- 1 <= target <= 10^9

- 1 <= nums.length <= 10^5

- 1 <= nums[i] <= 10^4

进阶：

- 如果你已经实现 O(n) 时间复杂度的解法, 请尝试设计一个 O(n log(n)) 时间复杂度的解法。

思路：

这题乍一看，是很经典的滑动窗口的题，感觉很简单。其实并非如此，还是有些区别的。主要在于题目要求的是“最小”的子数组，和一般的最大的子数组的要求不太一样。其实一种暴力的做法，也是不会漏掉答案的做法，是：我们遍历所有可能的起始位置，然后往后看，一旦找到和大于等于target的时候，就把这个长度记录一下，最后在所有的子数组的长度取个最小的（如果从某个位置出发，没有某个子数组的和符合要求，不妨把这个子数组的长度记为一个很大的数，这样后面取最小值的时候肯定也取不到它。但这是细节了）

上面的暴力做法，时间复杂度应该是\(O(n^{2})\)

一开始，因为想当然了，直接上来就是按照不定长的滑动窗口写。此时注意，一般写的不定长滑动窗口，都是针对求“最长”子数组的，所以一般我们是遍历终止位置，相应移动起始位置。比如，我们从索引0出发，窗口不断扩大，可能扩大到索引5的时候，发现子数组的和符合要求了，ok，记录更新一下结果，然后移动起点，再来继续判断。这个过程在求“最长”时当然没问题，但现在要求“最小”，不会有问题吗？因为，在上面的这个例子里，我们把起始位置移动到索引1的时候，终止位置已经在5那里了，此时即使这个子数组的和，我们判断符合条件，长度也已经是5了；但有没有可能，起始索引为1的时候，其实终止索引到3就可以满足条件了？那这样最终结果至多就是3啊？不会漏掉答案吗？
这题还真不会。因为，它规定了数组nums的元素全都是正整数。这就导致，滑动窗口往右扩大（在起始位置不变时），严格单调递增。所以再回到上面的例子，我们的担心其实是不存在的。假使真的说：起始索引为1时，终止索引为3的时候，就已经满足条件了，也就是1-3的这部分和，已经大于等于target了，那从索引0出发的时候，最多我们把终止位置遍历到3的时候，肯定也得满足要求了（0-3的和肯定大于1-3的和，所以肯定也大于等于target），这样我们第一次起始索引为0的时候，终止索引就不可能停在5那里。所以担心是不成立的。故这道题里，我们用一般的求最大子数组的滑动窗口的写法，也恰好能求出答案
