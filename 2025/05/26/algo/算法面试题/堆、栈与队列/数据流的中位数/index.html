数据流的中位数

Posted on 2025-05-26 In algo , 算法面试题

原题链接在这

题目描述：

中位数是有序整数列表中的中间值。如果列表的大小是偶数，则没有中间值，中位数是两个中间值的平均值。

- 例如 arr = [2,3,4] 的中位数是 3 。

- 例如 arr = [2,3] 的中位数是 (2 + 3) / 2 = 2.5 。

实现 MedianFinder 类:

- MedianFinder() 初始化 MedianFinder 对象。

- void addNum(int num) 将数据流中的整数 num 添加到数据结构中。

- double findMedian() 返回到目前为止所有元素的中位数。与实际答案相差 10-5 以内的答案将被接受。

示例 1：

输入

["MedianFinder", "addNum", "addNum", "findMedian", "addNum", "findMedian"]

[ [], [1], [2], [], [3], []]

输出

[null, null, null, 1.5, null, 2.0]

解释

MedianFinder medianFinder = new MedianFinder();

medianFinder.addNum(1); // arr = [1]

medianFinder.addNum(2); // arr = [1, 2]

medianFinder.findMedian(); // 返回 1.5 ((1 + 2) / 2)

medianFinder.addNum(3); // arr[1, 2, 3]

medianFinder.findMedian(); // return 2.0

提示:

- -10^5 <= num <= 10^5

- 在调用 findMedian 之前，数据结构中至少有一个元素

- 最多 5 * 10^4 次调用 addNum 和 findMedian

但最终是能通过一些测试样例，无法整道题通过，时间还是被卡了。但也可以理解，前面找第K个最大的元素的时候，它是给定数组，且只用找一次；而这里，数组是动态的，且可能会找多次中位数。虽然这个单次搜索第K个最大的元素的算法，时间复杂度是线性的，但毕竟数组长度在不断变大，加上变大之后也没有做什么额外的处理或维护，超时也是可以理解的
不过这里还有个细节要注意，就是，索引从几开始。我们实现的quick_select算法（也就是那个找出第K个最大的元素的算法）是默认从1开始的索引。要么是索引配合算法，要么是算法修改一下，配合从0开始的索引。这里是给所有的索引都加了1，配合一下算法（因为已经写好了，懒得改算法了……）

总而言之，这里用前面的算法，时间复杂度上是不够的。

这里参考了灵神的题解，感觉还是比较好懂的。首先，他是根据中位数的一种定义，也就是”能把数值集合划分成相等的两部分“这个定义入手的。因此，想到：如果我们能做到，维护两个集合，分别记作left和right，满足：left里的元素一定都小于等于right里的元素；
且二者元素数量接近，要么是相等的，要么是只差1的（可以规定，只差1的时候，我们希望left是多1的那个。当然，令right为多1的那个，应该也没问题）。如果能做到，那么找中位数就是很简单的了：判断一下left和right的大小，若相等，则此时是偶数个元素的情况，取left里的最大元素和right里的最小元素，求算术平均值，即得到中位数；而如果不相等，则应为left元素数比right元素数多1的情况，此时直接取出left里最大的那个元素就好

这里left和right，我们对它的要求是，能快速找到最大/最小元素，且还能做到可维护，也就是即使有新元素加进来，也能快速找到最大/最小元素（并且，我们并不要求一个集合能同时找到最大和最小元素，它只需要能找到一种我们指定的极值即可）所以，这里可以考虑使用堆结构。这样看，left应该就是一个最大堆，而right就应该是一个最小堆
并且，因为我们没有使用什么别的特殊功能，所以用cpp的容器里的priority_queue结构即可

现在我们再来看看，用priority_queue来做堆的话，我们能否在添加元素的时候维护上面的两条特性。首先，初始情况下，left和right应该都是空的，可以认为二者大小相等，此时一旦插入元素，根据我们的约定，应该往left里插入；且插入之后，就变成了left比right多一个元素的情况了
接下来看一般情况，假设left和right里已经有一些元素了，且满足上面的两个特性（这里有点类似数学归纳法吧），看看应该怎么维护：

如果二者相等，则插入之后的结果应该是left比right多一个元素。最直接的，如果直接往left里插入，会怎样？我们此时就无法保证left里的元素一定是都比right里的元素小的。换句话说，我们不确定这个新插入的元素是否是比right里的元素都小的。此时我们可以考虑，先把新元素插入到left里，然后我们找到此时left中最大的那个元素，加到right里，并且从left里弹出；之后再把right里最小的元素找到，拿到left里，并从right里弹出。最终，是可以满足两个特性的，也就是维护成功了

我们可以再细一点来看。先把新元素插入到left，之后找到left里最大的元素，加到right里，并从left弹出；此时left里的所有元素，一定是都小于right里的所有元素的，因为right此时的元素就两个组成部分，一个是它原来的（显然满足），一个是从left里找到并加进去的（因为是插入新元素后，left里最大的元素，所以也满足要求）。假设插入前，两个集合的元素数量都是k，则一通操作完了，现在left里是k个元素，right里是k+1个元素

接着，是找到right里最小的元素，拿到left里，并从right里弹出。那么，现在left里的元素依然是小于等于y里的所有元素的。因为，此时left里的元素就两个组成部分，一个是left原来的，它当然满足；一个是从righ里找到并新插入到left里的最小元素，因为是刚才的right里的最小元素，所以也满足要求。而此时，left里的元素个数为k+1，right里的元素个数为k。故，现在操作完之后，我们算是维护成功了

如果二者元素不相等，呃，也可以想见，是类似的分析方法。其实这个维护的手段应该还挺多的，想到一种就行，不算特别难。（主要还是用priority_queue，能直接获取到集合里的最大/最小元素，还是很方便的）

至此，我们能做到维护两个集合，且满足两个特性，故求中位数也是很容易的
