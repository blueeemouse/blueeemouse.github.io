Locomo

这里测试的时候，它不会去检索索引，而是直接进行精确的相似度计算了（那rag类方法显然是可以很方便地迁移过来） ## 评估指标： ### 回答： precision/recall/F1：主要是这个，对于一些直接给一个答案的，就是正常计算；对于多跳问题，因为会给出子答案，所以会计算每个子答案的F1然后求均值 0/1（对于二选一类问题） ### 检索： recall：看id来判断是否命中。分两种匹配粒度，一种是dialog，按照dia_id来匹配；另一种是summary，按照session来匹配。recall的计算和一般的没区别，就是 检索中命中的 / 有效的总数 # LongMemEval（长程对话问答） ## 评测指标： ### 检索： recall_any@k recall_all@k ndcg_any@k（这个和一般的ndcg没区别，感觉纯粹是为了和上面的指标对齐而已） ### qa： task-avg acc：这个是，先对每个类型的任务里的query的llm-judge求均值，得到每个类型的任务的均值指标，然后再对这些任务的指标求均值（虽然看起来和下面的很像，但是写写公式可以发现是不一样的） overall acc：这个和task-avg略有不同，它是直接把所有query的llm-judge结果拿来求均值 abstention acc：这类就是评估那些无法回答的问题，模型是否有识别出来，然后对这些问题计算llm judge的均值 --- # 类别：长对话里动态信息捕捉 # RealMem（ACL 26 findings）（这个起码代码比较齐全，感觉可测） ## 维度：长程，project states，schedules，personas（不仅是关注局部的信息了） ## 这个它有一点提得很好，就是qa timing，像LoCoMo还有LongMemEval，它们是after all sessions的，但RealMem就是Interleaved within sessions ## 指标上，它会评估检索内容和回答内容（也就是关注，有没有取到正确记忆，以及有没有真的把正确记忆用到回答里） ### 检索（会衡量准不准，好不好）： recall_any@k（top-k 检索结果里，只要命中了任意一个 gold session，就记为 1，否则 0；最后会对所有query求平均。它其实就是一种比较宽松的衡量检索效果的能力）

recall_all@k：衡量top-k结果里是否包含所有gold session，是则记为1，否则为0（是一种非常严格的指标）

ndcg_any@k：用NDCG衡量正确session在排序种的位置，正确结果越靠前，分数越高（所以它衡量的是排序质量好不好）

Mem_recall / average_mem_recall：这个也是衡量检索的记忆“好不好”，但不是像上面的指标那样侧重于lexical角度分析，而是从语义的角度，让llm判断，ground-truth memory里有多少条被retrieved memory覆盖了（取值范围0-1）（上面的recall是用session id来判断是否命中的，这里则是llm按照语义来判断）

Mem_helpful_score / average_mem_helpful_score：这个就是想用lexical来衡量也不太能做到了，可以说必须得上llm了。它衡量的是检索的memory本身的质量（而非说参考ground-truth memory），0表示检索记忆相互冲突/误导，1表示有一点帮助，2表示memory很有帮助，能比较完整支持回答 ### 回答： score / average_qa_score：这是一个llm-judge的指标，它会给单个query打分，从0-3（离散的），0表示回答和用户记忆冲突；1表示不冲突，但很泛泛，没有真正利用记忆；2表示用到了部分用户记忆；3表示像参考答案一样，充分使用了相关记忆（嗯，所以说可以认为它是在衡量回答，但衡量的维度侧重于“是否利用好了memory”，而不是单纯看对不对）

qa_score_distribution：这个就是评估上面那个指标的分数的分布，由此来判断系统的问题，是大多能充分利用记忆，还是说经常幻觉

qa_hallucination_rate：这个就是score=0的比例，也就是说回答和记忆冲突的比例（回答和检索内容冲突，其实就是这里所谓的幻觉了）

qa_perfect_rate：score为3的比例 # PersonaMem（COLM 25） ## 这个应该是主打用户偏好的变化吗？就是一个user profile既有静态信息也有动态信息 ## 想评估的llm的能力包括： ### 对user profile的记忆（偏重静态信息的记忆、提取） ### 对user profile的追踪 ### 生成个性化回复 ## 通过看代码，可以发现，它的评测是相对偏动态的。实现上，在一长串的对话里，它会在中间若干位置插入query，然后提问模型的时候，只会提供截至到query的上下文。且问的问题是时间顺序的，所以先问前面的，然后问后面的。那么每次模型见到的上下文就是越来越多的

===

偏向事实性

HaluMem（MemTensor的）（这个感觉真得测，挺常见的，而且确实是有人用过的）（可以说比较重要了，代码比较全面，整体也比较好；而且对于extraction的测评，正好用rag类的粗略chunk作为对比的话，可以体现memory类工作的一部分价值（如果它们那些复杂的extraction确实有用的话））（而且它评估的update的能力，很重要，值得关注，这是和一般的bench不一样的地方，加上如果我们想综合评估rag、memory、agentic search，可以从一个知识管理的统一角度来看待。而知识管理要评估的就得有这个更新方面的能力）

想评估的能力；

记忆提取（其实就是检索）

记忆更新（能否正确更新现有记忆，保持一致性）

记忆问答（就是说最后效果好不好，是一个综合能力的体现，既要提取好，也要更新好，最后还要利用好检索内容）

数据构成：

记忆分了三类：persona memory（比如个人兴趣等）、event memory、relationship memory

设计的问题种类：

基础事实回溯（Basic Fact Recall）：直接询问对话中的客观事实。 多跳推理（Multi-hop Inference）：需要整合多个对话片段进行逻辑或时间推理。 动态更新（Dynamic Update）：测试系统追踪信息随时间变化的能力。 记忆边界（Memory Boundary）：询问对话中未提及的细节，测试系统是否会凭空捏造。 泛化与应用（Generalization & Application）：基于已知偏好在场景中给出建议。 记忆冲突（Memory Conflict）：故意提供错误前提，测试系统能否识别并纠正矛盾。 ## 评估流程： ### 三类任务：memory extraction，memory update，qa ### 逐session添加到memory里的，而且它实验里也没有测试rag类方法。那比较麻烦了，可能只能自己做一些假设了（比如不建立索引了，直接存向量……然后设计一些简单的驱逐/添加策略，像Evo-Memory那样） ### 指标： #### memory extraction： （首先要注意，这里是衡量extraction效果，也就是从对话里提取出要存储的信息的效果衡量，既要衡量准不准，也要衡量是否把重要的东西都提取出来了，那就对应了acc和recall了） memory_integrity：这里我们会给检索记忆进行打分，判断memory的完整性（就是指该检索出来的东西是不是都检索出来了，所以实际计算的指标还是recall）（0-2离散分数，llm判断。给llm的输入是我们检索到的memories以及golden memory），然后算得分为2的比率，作为recall。虽然是0-2，但是实际计算的时候会乘上0.5用以归一化 memory_accuracy：这个就是衡量我们检索的memory是否足够准确，给模型的输入是完整对话+golden memory+单条我们检索到的memory。这里llm会打0-2的分，同时也有一个bool变量，is_included_in_golden_memories（这里主要是 FMR（False Memory Resistence）： F1：其实就是对recall和acc的一个综合度量，和一般f1定义没区别，二者的调和平均 #### memory update： （其实memory update和qa，最后的指标都是统计一些比率。这些都是用llm给每个query的回答进行打分，看是属于哪一类，然后统计比率） correct_update_memory_ratio（生成记忆完整覆盖目标更新内容，关键字段正确，且旧记忆被替换/过时化） hallucination_update_memory_ratio（有相关“更新”但事实错误或与目标更新冲突） omission_udpate_memory_ratio（没更新，或更新了但缺关键内容（部分遗漏）） other_update_memory_ratio（其余异常） （由此我们也可以看到，虽然我们最终效果上可能还是看correct的比率，但它这里定义了一些错误的类型，因而可以辅助判断update环节到底有没有幻觉，幻觉严不严重） #### qa： correct_qa_ratio hallucination_qa_ratio omission_qa_ratio # BEAM（BEYOND A MILLION TOKENS: BENCHMARKING AND ENHANCING LONG-TERM MEMORY IN LLMS，ICLR 26）（看起来它评估的更多是llm自己的能力啊？） ## 想评估的能力： 拒答（Abstention）：当证据缺失时，模型是否能拒绝回答。 矛盾解决（Contradiction Resolution）：检测并调和对话中相隔较远的矛盾陈述。 事件排序（Event Ordering）：识别并重建对话中信息的演变顺序。 信息提取（Information Extraction）：从长历史中召回实体和事实细节。 指令遵循（Instruction Following）：在长上下文中持续遵守用户指定的约束。 信息更新（Information Update）：随着新事实的出现修正已存储的事实。 多跳推理（Multi-hop Reasoning）：整合多个非相邻对话片段的证据进行推理。 偏好遵循（Preference Following）：根据用户演变的偏好进行个性化响应。 摘要（Summarization）：抽象并压缩对话内容。 时间推理（Temporal Reasoning）：对显式和隐式的时间关系进行推理。 ## 数据构成： ### 100场长对话，token跨度涵盖100k、500k、1M以及极端的10M tokens；总共2000个问题（每场对话对应20个问题） ### 涵盖19个领域，包括coding、math、health、finance等 # MemoryAgentBench（Evaluating Memory in LLM Agents via Incremental Multi-Turn Interactions，ICLR 26，314stars了，挺高质量的感觉） ## 想评估的能力： ### 精确检索（能不能检索到关键信息） ### 事实整合（检索到多种，甚至冲突的信息，如何整理） ### test-time learning，类似于in-context learning，或者说是对检索内容的利用效果 ### 长文本理解与推理 ## 这篇其实比较关键，它应该也有很多可以参考的点。主要是它好像在qa数据集上也测试了类似agentic search以及memory的方法。不过它本身并没有提出一个新的bench，更多是类似于用多个现有bench进行综合评估 # ES-MemEval（WWW 26） ## 想评估的能力： 信息提取（Information Extraction）：识别会话内和跨会话的关键事实。 时间推理（Temporal Reasoning）：推断事件的时间顺序和因果关系，追踪用户状态的演变。 冲突检测（Conflict Detection）：检测并解决记忆中的矛盾，以保持与用户当前状态一致。 拒绝回答（Abstention）：在信息不足时拒绝回答，以确保可靠性。 用户建模（User Modeling）：推断并更新用户的特质、偏好和状态。 （没有太特别的维度） ## 数据构成： ### 任务上主要是三种，qa（评估跨会话信息检索和回答）、summarization（分析跨会话信息和用户动态变化）、dialogue generation（模拟真实交互，反映模型在个性化情感支持中的综合表现）

想评估的能力：

经验蒸馏与存储（Distillation & Storage）： 智能体能否从早期的行动和环境反馈中提取关键信息，并将其转化为持久记忆。 跨会话的因果推理（Cross-session Causal Reasoning）： 智能体能否识别并利用前序会话中产生的潜在约束（如兼容性要求、共享偏好、中间推理结果）来指导后续决策。 长程任务执行（Long-horizon Execution）： 在长达数十步的操作中保持任务状态的一致性。 ## 数据构成： ### 包含766个任务，平均每个任务包含 6.9 个子任务（会话），平均每个任务需要 57 个动作步骤 ### 涵盖的场景方面，有bundled web shopping、group travel planning、progressive web search、sequential formal reasoning（和常见的qa形式不太一样，其任务相对更复杂、多阶段） ---
