memory随笔

Posted on 2026-03-09 Edited on 2026-05-04 In 感想与思考

memory的设计似乎是要分场景的，不知道现在有没有做统一的、通用的memory架构设计的

memory周期性更新、快慢记忆/系统，感觉有点标配的意思（起码在插件式mem里是这样的）

检索，不是说基于相似性的检索就不行了，但只有它是一定不够的，而且尤其要看场合，像日常的长期对话这种，它可能就还好，如果是对知识、模式等要求比较高的场景，还用这一套就不太行了。设计知识的结构（比如加入utility这种属性）、引入llm自己来选择检索，都是可以考虑的思路

其实memory感觉上可以用到很多的垂域。回想一下，本来RAG就是用来解决llm应用到垂域效果不好的问题的，如果微调又担心灾难性遗忘。这一套故事，或者说这些问题，是垂域应用里绕不开的。但memory在这个场景下相比RAG的本质好处是？大家为什么用memory而抛弃RAG呢？是觉得维护向量数据库很麻烦？

嗯，外挂式的memory retrieval和llm内生（训练得到）的retrieval能力，哪种更好？又或者说二者都需要？

memory架构、组织形式，这种真的有意义吗……（感觉要明确目的，不能是觉得“这个架构合理”就用了。例如，想避免rag里检索内容碎片化，所以想引入一些特定形式）

引入结构，可能就更新不方便；不引入结构，纯向量，检索可能碎片化。然而在尝试引入结构的时候，一旦有类似于构建“边”“关系”“联系”的操作的时候，感觉开销总是小不了

还有，总结一下见到的memory的结构，本质上可以说，都是“原始数据-->高层抽象”这一套，只不过具体有几层、如何构建这个抽象，可能大家手段上有细节差异

得找到一些经典的、用得比较多的baseline，跑一跑，才能有更多感想吧

流式memory，类似MemAgent的那种，其实都可以视为短期记忆的一种实现。有意义，但应该不够（最好有些实验证明）

这类流式memory，好像一些是训练llm让它学会更好提取（可能用rl之类的，比如MemAgent），一些是说参考RNN的思想，把信息压缩到一个固定大小的表征里（这时可以用一个最优化的视角来看它，因为这相当于一个压缩问题，我们希望在固定的size下尽可能把有用信息压缩进去）

这种流式memory，也可以考虑和检索式memory结合？二者搭配着来，类似于多级cache，流式memory里存储常用的，而检索时memory就是比较大、但比较慢的那级cache（不过其实思想上现在也有一些memory架构是这么做的了）

其实可能是必须要这种的。因为llm的架构决定了，随着token的不断增加，前面的token就是会被稀释（既有position bias的原因，也有注意力机制的原因。最简单的，我们生成下一个token的时候，要聚合前面的所有token信息，token越来越多，那即使有一个token特别的重要，可它的权重多多少少会被稀释）。这种情况下，外挂式memory检索得再好，也还是无能为力的。所以此时working memory就可以派上用场了。比如说kv cache层面的，我在聚合信息的时候去强调我们的关键信息，这可能就是有用的。或者说，当关键信息本身就特别长，以至于长到快逼近context window乃至超过context window了，外挂式就无法本质上解决问题）

光想是没用的，一定得有一个好的场景、好的bench才行

WebArena（网页导航智能体），这种就不错，因为在真实网页里执行一些复杂任务，足够难、足够长程）

llm/agent+memory，确实可能可以在一些垂域做得很好。但细想一个问题是，如果我们不能通过微调给模型参数注入知识，那么模型对这个垂域的了解几乎都是来自检索。这里有个问题，就是一个知识只有检索到了，模型才知道；下一次没检索到，它可能就还是不知道。但人不是这样。它是，一开始可能什么都不懂，用得多了，会渐渐理解，开始记得一些、理解一些，之后可能不需要检索它也知道了。这样，长期下来有可能发生一些更高层次的理解，从而更有insight；但模型+memory好像还是做不到这样

正好，看到一篇Internalizing Meta-Experience into Memory for Guided Reinforcement Learning in Large Language Models（ustc的），它的做法某种程度上是互补的。即，它要求模型实现进行一些实验，得到一些结果，然后从这些结果里蒸馏出经验，最后把这些经验训练到模型的参数里。这是类似于事前工作，而我们的memory则可以在事中进行补充

此外，这篇论文里提到的一个观点是，检索式地使用经验，会导致推理的时候上下文变长，从而使得推理更慢。这显然也是一个efficiency上值得考虑的点。其实也已经有一些工作在这方面下手了，不过好像没看到有从kv cache这一块下手的，但是有那些把memory给压缩的，也是一条路。而把经验训练到模型参数里，一个好处就是不需要显式给文字经验了，推理更快了

memory，agentic search，rag，区别在哪里？

感觉一个是场景上的区别。memory它设计之初就是在使用的时候动态更新的，它一定得适应流式的输入，而且要能不断更新然后供给使用。所以才有各种设计memory准入、更新的工作（因为要流式），还有设计memory压缩的工作（因为流式里是无限多的输入）。rag它一个重要的应用场景就是本地知识库，本地知识库的更新相对而言没那么频繁、没那么庞大，所以对更新其实没那么高的要求，也不太会担心“无限多输入”这个情况。而agentic search，它只是一个技术，可以应用到memory，也可以用到rag

memory的输入是不断的、动态的，而rag的输入一般是大批量、一次性的。这也是一个区别。或许正是这个场景上的区别，让二者的技术有所差别。可能我们也不需要纠结到底哪个名词对应哪个技术，根据我们的场景，自然的会选择响应的技术，效果好就行了

要说技术上的区别，agentic search可以说是一个思想、一个工具，它用到rag里，就成了agentic rag；用到memory里，那我们的memory retrieval就是agentic retrieval了（现有做memory的，有在研究memory组织架构的，有在研究memory evolution的，有在研究memory efficiency的，这些都和memory retrieval不冲突，乃至可以结合）

说一些个人观点，memory组织形式，似乎不应该投入大量时间研究。一个是大家自己玩自己的，对比其实很不方便，也很难有一个统一的形式；二来这个也比较的工程，而且比较依赖直觉（类似于，我觉得这样能利用更多信息，就这样设计了）；再来就是它和rag其实没啥差别。相比之下，memory evolution（我自定义）、memory efficiency、memory augmented reasoning、streaming memory，这些就比较plug-and-play，更加泛用

latent memory，这个形式我们是不是也可以认为是ai-native的？这个形式就不是给人看的了，而是专门给模型看的。相较而言，它更compact，但还有其它好处吗，比如更有利于模型推理？

kv层面的东西，算是latent memory吧？而且这样感觉是计算量比直接检索文本然后送到上下文里更小的（这是老一辈已经想到的东西了啊，LongMEM，23年nips就已经有了。这篇感觉做得不错）

但别说，kv这种形式的memory，它不可避免会对推理有一些干扰（如果不微调的话。因为相当于我们推理过程中，聚合过往历史信息之后的token表征会发生变化。固然直观上我们觉得这是有利的，但没有实验之前都不好说。相比之下，如果是检索文字然后放到上下文窗口，这种方法它虽然计算量更大，可是不会损害模型的能力）

这类latent memory不方便编辑是一个问题吧。相对的，token-level的明文记忆，编辑起来就方便多了。所以，构建一个多种介质的memory系统可能是关键（现在到底有没有人做呢？值得看看）

memory应该和推理相互促进。更进一步说就是检索应该是和推理相互促进的

这是在rag时代就已有的，现在虽然演化出memory，但核心思想还是一样。检索的结果应该帮助推理（这是最本质的出发点，但不是说仅仅用于补充上信息的不足，而是说补足gap之余还能推动推理。比如形式整理得更好？就好比说一道题，两种写法，一种是非常跳步的，一种是详细的、娓娓道来、有注释的，即使答案都是对的，显然也是后者对我们的启发更大）

体现在做法上，就比如说我们用rl进行一些训练，要求加上了检索内容之后推理效果提高（这有很多衡量方法，最简单是看reward/答案，细粒度一点可以中间给一些credit……）

而推理也应该帮助检索，这主要是因为检索需要依赖于我们的推理状态（context）

这方面（指memory帮助推理还是有一些工作在关注了吧）

这其实可能也可以从in-context learning的角度去看。就算检索到了内容，能不能利用好、并生成正确合适的回答，也是问题。memory和in-context learning其实是正交的

对memory的评测，得来点类似压力测试的感觉，把context增长再增长，才能看出来一点真正的效果吧

感觉实际使用agent的时候，得让它自己自主一点去注意context的使用情况吧，就算说不是让它自己就自动进行compact了，起码快满了的时候它也应该提醒我，然后来问我要不要准备compact之类的。现在这样纯靠人自己感觉是不是快满了，也太智障了吧

现在memory方面的工作是不是大部分聚焦于事实性信息，一些最近的工作开始聚焦于过程性经验的沉淀和检索？但有把两个结合起来的工作吗？

话说它们评测的时候是会实时沉淀出traj/exp并存起来吗？还是说又是提前跑出来再在推理的时候检索？

而且过程性经验的检索，可能才需要utility、score之类的指标。如果只是factual information，那其实没有所谓重要性一说，纯粹看什么时候问到了。这是一个区别

以及，目前有同时做procedural memory和factual memory的框架/方法吗？

procedural memory的可迁移性，也值得稍微关注一下。这是procedural memory相比factual memory独有的问题

latent memory形式，有没有必要？是否需要一些验证？

这主要是因为我们看一些cli agent的应用的时候，它们都是直接加载明文的

llm没有状态，所以我们需要memory；但即使有memory，即使上一次检索过了，下一次我们检索到相同的内容，模型还是要重新处理。相比之下，人脑是有状态的，可能上次见过某个东西，下一次就知道了；即使有点遗忘了，但再次看，我们处理、理解的效率和效果也有所提高（“书读百遍，其义自现”）。对应过来，就是kv cache的复用？

      1. 12.1.1. 体现在做法上，就比如说我们用rl进行一些训练，要求加上了检索内容之后推理效果提高（这有很多衡量方法，最简单是看reward/答案，细粒度一点可以中间给一些credit……）
    2. 12.2. 而推理也应该帮助检索，这主要是因为检索需要依赖于我们的推理状态（context）
    3. 12.3. 这方面（指memory帮助推理还是有一些工作在关注了吧）
    4. 12.4. 这其实可能也可以从in-context learning的角度去看。就算检索到了内容，能不能利用好、并生成正确合适的回答，也是问题。memory和in-context learning其实是正交的
  13. 13. 对memory的评测，得来点类似压力测试的感觉，把context增长再增长，才能看出来一点真正的效果吧
  14. 14. 感觉实际使用agent的时候，得让它自己自主一点去注意context的使用情况吧，就算说不是让它自己就自动进行compact了，起码快满了的时候它也应该提醒我，然后来问我要不要准备compact之类的。现在这样纯靠人自己感觉是不是快满了，也太智障了吧
  15. 15. 现在memory方面的工作是不是大部分聚焦于事实性信息，一些最近的工作开始聚焦于过程性经验的沉淀和检索？但有把两个结合起来的工作吗？
    1. 15.1. 话说它们评测的时候是会实时沉淀出traj/exp并存起来吗？还是说又是提前跑出来再在推理的时候检索？
    2. 15.2. 而且过程性经验的检索，可能才需要utility、score之类的指标。如果只是factual information，那其实没有所谓重要性一说，纯粹看什么时候问到了。这是一个区别
    3. 15.3. 以及，目前有同时做procedural memory和factual memory的框架/方法吗？
    4. 15.4. procedural memory的可迁移性，也值得稍微关注一下。这是procedural memory相比factual memory独有的问题
  16. 16. latent memory形式，有没有必要？是否需要一些验证？
    1. 16.1. 这主要是因为我们看一些cli agent的应用的时候，它们都是直接加载明文的
  17. 17. llm没有状态，所以我们需要memory；但即使有memory，即使上一次检索过了，下一次我们检索到相同的内容，模型还是要重新处理。相比之下，人脑是有状态的，可能上次见过某个东西，下一次就知道了；即使有点遗忘了，但再次看，我们处理、理解的效率和效果也有所提高（“书读百遍，其义自现”）。对应过来，就是kv cache的复用？
