search-o1

最主要的贡献就是首次把agentic search的能力结合到了推理模型中（所以重点是结合到推理模型吗？在此之前是已经有研究把agentic search能力结合到llm里了吗）

这里的agentic就是说，可以在单次推理任务中多次、迭代地按需触发检索

还有就是，有一个独立的Reason-in-Document模块，从而缓解检索噪声的问题（就是说把检索的内容做一个精炼），同时也是为了让推理的过程保持连贯

体现在输入上，它的输入是查询的query、检索内容、已有的推理链（也就是让这个模块明白现在推理到哪里了）；输出则是精炼之后的推理步骤，这是可以直接接再原来推理链后面的，可以引导推理（相比于直接把检索内容接到原来的推理链后面，应该是更好的）

这一步也是调用一个llm来实现的，就是回答的那个llm。但它强调的点是，这里的精炼是单独的一个步骤，不会和主推理流程耦合

但这篇它竟然并没有说进行什么训练，它的实现就是靠特定prompt而已，只不过多了一个单独处理检索文档的部分，这个部分也没有微调一个模型，只是独立出来了 # Atom-searcher 研究的是agentic deep research，姑且也算到agentic search里吧（但后续要不要它，可能还得细分）（我去，它还搁着刷HotpotQA之类的qa benchmark的分吗，无敌了……不是deep research吗）

主要的动机还是说，这类研究，一般都是用outcome-based RL，那么就存在一个问题（论文说是两个，感觉实质是一个）：仅参考结果，那么奖励粒度过粗了；如果中间有对的步骤，却因为最后生成答案错了，导致被判错，这是一种误导，阻碍模型学习中间的有效策略

论文方法：

原子化思维（Atomic Thought）： 将复杂的推理过程分解为细粒度的功能单元（如 <Reflection>、<Verification>等），使推理过程更清晰。

原子思维奖励（ATR）： 利用推理奖励模型（RRM）对这些细粒度单元进行评分，提供中间监督信号，缓解梯度冲突。

课程启发式奖励调度（Curriculum-inspired Reward Schedule）： 在训练初期优先考虑过程奖励（ATR），随着训练进行逐渐转向结果奖励。（它的奖励是一个加权求和，即结果的奖励和过程的奖励。初期过程奖励是比较高的，但这个奖励的权重会随时间衰减（有一个自定义的函数））

这里可能的insight在于，初期模型的答案一般不对，所以此时奖励是尤为稀疏的（如果还是靠结果的话）。那么这个阶段就重点看中间过程，类似于“手把手”带着；后期，则更多是为了提高答案的准确性（可以说这是我们的一个目标），同时也是防止模型出现reward hack（因为它可能会为了赚到中间过程的奖励，而生成一些无用的推理过程，但最后答案生成不对）

关于训练：

比较经典，先是一个sft，让policy model掌握特定格式（而这里的训练数据一般都是符合要求的traj，通常是用一个现成的大的模型来得到的）；之后是rl（论文是用grpo），让policy model进一步学会掌握这些动作，提高最后的效果 # Agentic Reasoning: A Streamlined Framework for Enhancing LLM Reasoning with Agentic Tools 这篇论文主要是想把外部工具集成到模型推理中，也就是想让模型学会何时调用这些外部工具。只不过这里的工具是一个个的agent（而非一个api）

所以主要的亮点变成了这些agent了……调用agent确实有一些好处吧，任务中间的内容是给这个agent处理，而不需要给这个回答的llm，所以它不会占用回答的llm的context；此外，毕竟agent执行有自主能力，效果一般也比api会好一些（虽说开销也更大）

另一个小亮点是，它调用的工具里，有一个所谓mind-map，也就是会对当前的推理链条进行一个总结，保存成一个kg，后续推理的时候会进行参考，由此能保证推理的连贯性（虽说为什么要保存成kg也存疑，而且真有效果吗……）

更重要的是，论文也没有什么训练，就是纯靠模型自己的能力…… # Toolformer 这篇它本身是聚焦于教会模型使用外部api工具，不算是聚焦于search。不过本质是相同的，现在agentic search其实也有点像教会模型使用外部搜索工具，且要求检索内容不要干扰模型推理

现有方法的不足（果然是早期，还有人手工标注）

作者指出，现有让模型使用工具的方法主要存在两个局限性：

   依赖大量人工标注： 许多方法需要人类手动标注模型何时以及如何调用工具，这不仅成本高昂，而且人类认为有用的调用方式未必是模型认为最有效的。

   局限于特定任务： 现有方法往往将工具使用限制在特定的设置或任务中，缺乏通用性。

论文提出的方法如何解决问题（同样也可以看到，和现在遍地走的rl不太一样）

论文提出了 Toolformer，这是一种以自监督（Self-supervised）方式学习使用工具的模型。其核心流程如下：

   采样 API 调用： 利用模型在上下文学习（in-context learning）中的能力，通过少量示例引导模型在大量文本数据中生成潜在的 API 调用。

   执行 API 调用： 运行这些 API（如计算器或搜索引擎）并获取结果。

   过滤 API 调用： 这是关键步骤。模型会计算添加 API 调用及其结果后，是否能降低预测未来标记（tokens）的损失（loss）。只有那些真正能帮助模型更好预测文本的调用才会被保留。

   微调： 在保留了有用 API 调用的增强数据集上对模型进行微调。

在过滤阶段，模型比较两个损失值：

   Li+​：提供 API 调用及其结果时的加权交叉熵损失。

   Li−​：不提供 API 调用，或提供调用但不给结果时的最小损失。

只有满足 Li−−Li+≥τf​（即损失降低程度超过阈值 τf​）的调用才会被采用。

注意，Toolformer 的 API 种类在训练前是固定且预定义的。作者为每种工具提供了少量的演示（demonstrations），然后让模型在海量语料中自动寻找这些工具的使用场景。论文中具体研究了五种工具：问答系统、计算器、维基百科搜索、机器翻译和日历。在训练完成后，模型只会调用它在微调阶段学习过的那些 API。这是因为 Toolformer 的核心原理是学习如何生成特定的 API 调用标记（例如 [Calculator(args)]）（所以可以说，本质上还是加几个新的特殊token让模型学会生成） # Search-R1: Training LLMs to Reason and Leverage Search Engines with Reinforcement Learning 用rl教模型学会何时使用搜索引擎（包括动态的生成query）。这里的故事、动机没有什么新颖的，方法上大体也是那一套（outcome-based reward，multi-turn search interactions）

除此之外，可能就是多了一些对于rl，llm模型选择，response length dynamics的一些经验性insight吧

btw，解释一下，标题说是search engine，但其实不局限于那种我们一般理解的搜索引擎（Google，bing，etc）。也可以是常见的本地dense retriever（也就是向量检索）。事实上，论文做实验，依然是在各个QA数据集上做，而它实验用的也的确是local dense retriever
