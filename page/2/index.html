Code Graph Model (CGM)：A Graph-Integrated Large Language Model for Repository-Level Software Engineering Tasks

Posted on 2025-07-15 Edited on 2025-07-16 In 论文阅读 , nlp , llm , llm4code

Venue：arxiv

date：2025-06-23

动机：

感觉动机这一块其实挺常见的。llm发展很快，将其应用到单文件的各种代码任务都表现得很好，但是到了更加真实复杂的SE task上，例如repo-level的任务上，它的表现就不够好了

而现有的方法，表现更好的是利用闭源模型的基于agent的方法，但这类方法，因为依赖于闭源模型，所以难以接触到模型权重，因而难以进行改进；同时基于agent的方法，通常推理步骤会很长，因此中间的错误可能会不断积累导致影响决策
所以，为了避免这些问题，我们会想用开源模型，并且避免用agent的方法。但现在基于开源模型的方法，性能还是比上面提到的那类要差一截，还不太能处理repo-level task，因此是有极大的改进空间的
而论文尝试做出的改进，则是想让模型对repo的理解更上一层楼，具体来说是让模型更加了解repo里的代码文件本身的含义，以及相互之间的关系

insight：

怎么乍一看，没什么新奇的呢……也是想提高模型对repo的理解（语义以及结构），也是用graph结构，也是让llm利用graph，顶多是细节上不太一样，框架没怎么变化吧（和RepoGraph对比一下吧……）

contribution：

提出了一个新的架构，大概是在开源的llm上加一些新的模块，用来提高llm对repo的理解，然后可以一起进行微调，从而适应SE tasks，且性能不错（最优的结果还是加上了Graph RAG。但话说，这个设计的Graph RAG是真的出于锦上添花，还是说，本来仅仅用新架构，无法work，或者说性能不够好，所以特意设计了一个Graph RAG来make it work？可能得看一下实验表格里有没有单独用新架构而没用Graph RAG的结果了……）

配套的，还设计了一个Graph RAG的框架，可以进行辅助

SWE-BENCH：CAN LANGUAGE MODELS RESOLVE REAL-WORLD GITHUB ISSUES？

Posted on 2025-07-14 Edited on 2025-07-15 In 论文阅读 , nlp , llm , llm4code

Venue：ICLR 24

date：2024-11-11

动机：

现在大模型发展过快，我们对它们的能力的评价有点跟不上了（如果还是像以前那样，在一些固定的榜单不断把分刷上去，似乎效果提升并不大，因为各家公司推出的新模型，通常都会在榜单上有若干项刷到sota，然而使用体感上好像没有那么大的提升）
需要一个真正有含金量的benchmark，对llm有难度，且具有实际意义，才能更好地衡量llm目前的能力边界
而现实生活中的软件工程，就是一个不错的测试场景。它具有实际应用价值，同时因为软件工程内部各个部分都相互联系，关系错综复杂，因此对llm的长上下文处理能力、复杂推理能力等都有很高的要求。如果llm能很好地在这个场景进行应用，那无疑表明llm的能力有了实质性的提高
因此，文章提出了SWE-bench，收集了若干GitHub仓库的issue，测试llm能否解决这些issue

insight：

这篇论文主要的工作还是提出高难度的benchmark。这可能更多说明了，高质量的工作未必一定就是算法层面的创新，如果确实注意到了某个领域里有效的benchmark的缺失，那么构建一个solid的benchmark，无疑就是一个极有影响力也非常有意义的工作

contribution：

提出了新的、高难度的benchmark，可以在长期衡量模型能力的进展

微调了一些模型（但意义可能更多在于说明，即使微调之后效果也还是不够好，未来还有很长的路要走）

RepoCoder：Repository-Level Code Completion Through Iterative Retrieval and Generation

Posted on 2025-07-13 Edited on 2025-07-15 In 论文阅读 , nlp , llm , llm4code

Venue：EMNLP 23

date：2023-10-20

动机：

现有的代码补全工具，通常仅能利用上单个文件里的上下文信息进行补全。但如果要进行repo-level的补全，那就不能仅仅依靠单个文件的信息，而应该对整个仓库的上下文信息有所选择地利用（因为仓库里的文件相互之间是有相较于单个文件更为复杂的依赖关系的）。因此需要专门提出repo-level code completion这个任务进行研究

现有的研究这个任务的方法，主要是两类，
一类是依赖静态代码分析与启发式规则的方法，另一类是基于有标签数据，对语言模型进行微调的方法

第一类，它能可靠地解析上下文，但是补全能力首先，无法生成不同长度的内容，且不支持再文件任意位置进行补全（说到底，可以认为是rule-based的通病）

第二类，主要问题在于泛化性不够好，对于没见过的仓库，要进行补全，效果还是比较差的

insight：

但凡是repo-level的任务，不管是代码理解还是代码生成，抑或是其它的，应该都要求对repo有一个整体的理解，以及在处理片段代码的时候能利用好、检索到正确的上下文。这篇论文最核心的贡献可以说就是在解决这个问题：如何更好地得到正确的、合理的上下文。它的做法是改进RAG，变成多轮的RAG，以期提高检索质量（联想一下之前被的REPOGRAPH，它是通过把仓库的各文件之间的关系抽象为一个graph，用处理graph的方法来处理文件之间的关系，并由此得到正确的上下文。二者似乎是可以融合的。不过这也挺显然的吧，毕竟人家REPOGRAPH设计之初就是一个plug-in的模式，当然可以融合。但更重要的可能是，二者的思路不同，一个是利用外部的模型进行辅助，一个是把数据的形式进行转变。两条思路是并行的，且可以合并的吧）

另外，这个由单轮的RAG变成多轮RAG的迭代的方法，也是一个比较通用的方法，或者说技巧？虽说也有可能因为检索到的结果不够好，反而影响了补全效果，然后下一次迭代时，搜索得到的结果更差，形成负循环，但这也是有改进余地的吧（但话又说回来，看论文2.2 Code Retrieval部分，它也专门提到了，基于之前的补全结果来检索，效果就是能有所提升，即使说补全结果不那么准确。虽然不清楚这个结论的依据是什么……）

还有，用滑动窗口把一个大的内容，分割成若干个小的内容，似乎也是一个实践上常用的技巧

contribution：

提出了一套新的框架，RepoCoder，是一种迭代地使用RAG的框架，可用于repo-level code completion task

提出了一个新的benchmark，RepoEval，可以衡量模型在repo-level code completion方面的能力。且包含了多种粒度

对RepoCoder进行了严格的分析，便于后续的改进

REPOGRAPH：ENHANCING AI SOFTWARE ENGINEERING WITH REPOSITORY-LEVEL CODE GRAPH

Posted on 2025-07-10 Edited on 2025-07-16 In 论文阅读 , nlp , llm , llm4code

Venue：ICLR 25

date：2025-03-18

动机：

实际软件开发里涉及的代码的体量较大，是repo-level的，而非一般的function-level或file-level。虽然现在有各种coding llm，在file-level及以下的各种任务（如补全、debug等）都做得比较好，但迁移到repo-level任务上表现欠佳
原因在于repo-level的代码文件之间有复杂的依赖关系，因此整个repo内部是有结构的，要在repo-level的任务上表现好，就需要能掌握这种结构。现在的各种方法对这个结构没有很好地把握住
因此考虑提出用graph结构来更好解读、利用这种repo里的结构

insight：

用graph结构来定位和理解。把代码之间的关系抽象成graph，其实是稍微会好一点的，比起直接把所有的文件一起当作输入

contribution：

提出了一种plug-in module，可以适配现有的方法，能增进对代码仓库的整体理解以及定位
