Untitled

Posted on 2025-05-11

abstract

现在的cv技术，因为它是数据驱动的，所以其实它是从大量数据中“推测”出物理规律的（这种方式不够显式，进而导致可解释性等不足）；而传统的cv技术，虽然是明确有物理规律支撑的，但性能又不够好，比不过深度学习的那些技术。那么，弱国能把物理模型与深度学习的技术融合，或许就能结合二者的优点，既有高性能，又有可解释性、泛化性、可证伪性。而这可以认为是把物理模型当成数据驱动过程中的一个约束条件，也就是我们约束深度学习模型必须满足物理规律（相当于一种融入先验知识的方法）

这篇文章就是在讲一些把物理模型和深度学习模型给融合起来的技术，也即所谓physics-based machine learning

intro

cv其实和物理联系很紧密，它不像nlp那样都是符号体系，它是确实和物理世界的规律相关的。很多任务都可以归结为物理的问题。

但物理和cv的结合并不是研究的重点（也就是没有受到重视），比较物理的传统方法也被数据驱动的深度学习方法所超越。但深度学习方法还是有它的问题，诸如幻觉。而且想量化深度学习方法的误差也不容易。另外，如果是想从ml理论角度来解决深度学习的问题，也比较难，现在的ml理论还是太空中楼阁了，作出了过多的假设，然而现实中未必能做到

现在，要做的事很明确了，就是尝试一下能不能把物理方法与数据驱动的方法结合起来，这样或许有可能得到一个结合了二者优点的方法（能使用少量数据，就达到很好效果，且可解释性强）。然而现在的问题就是，怎么结合？不妨从深度学习的pipeline入手，也即模型，损失，输入，输出，等等

when to use physics-based learning

这一节就是字面意思，来研究什么时候会应用到physics-based learning

一些问题里，用物理模型可以直接很好地解决；另一些问题里，它完全无法用物理模型来描述或量化，因此只能用深度学习的方法。这两类都不适合用physics-based learning

而有一些问题属于第三类，也就是，它们的物理模型是不准确的，或者物理模型里的参数是不知道的，因此不能直接用物理模型来解决，这就是第三类问题，也就是适合用physics-based learning，即，把物理模型当成是一种inductive bias，放到深度学习里

总的来说，评价是否适合用physics-based learning，就看是不是这种场景：物理模型有一定的意义，但光靠物理模型无法很好地解决（推理）问题。这里涉及到三个评价因素，即数据的好坏，物理的好坏，以及把二者结合起来的难易程度

关于数据和物理的好坏，我理解的是，这里可能是想说，两种方法的相对效果（从而我们可以知道，物理模型是不是不能很好地解决问题）。最直接的评判方法就是，比一比两种模型的性能。但可能存在这种情况，就是物理模型不能直接给出结果，所以可以考虑另一种方法，就是，用深度学习的隐表示，能不能推导出物理规律（这个怎么理解？）；最后一种方法是，找一个中间任务，确保物理模型也能直接输出结果，再比较两种模型在中间任务上的效果

how to use physics-based learning

接下来会从三个角度来阐述如何把物理模型和深度学习模型结合起来

1.数据集角度

其实也就是把符合物理规律的数据拿去给模型训练，从而让模型潜移默化地学习到物理规律（是一种inductive bias的形式。比如，给一些平抛小球的轨迹数据，以及初始的速度，要求模型学会预测轨迹；这些轨迹数据是符合重力的，那模型学完之后应该就能隐式地掌握到“重力”这个概念了）

但用到合成数据，就必须要注意和真实数据之间的gap（也就是让合成数据尽可能真实）

2.网络架构角度

这里介绍了两种技术，一个是residual physics（一个典型例子是，预测抛物线轨迹。纯深度学习方法就是获取一大批数据和起始条件，让模型去预测；而residual physics则是，先用物理规律得到一个大概的预测值，比如我们用一个理想的模型，没有任何阻力的抛物线方程得到一个预测值，之后我们让深度学习模型去预测残差，也就是因为各种现实因素导致的物理模型预测值和现实值之间的差值，例如现实里有空气阻力，空气阻力有可能涉及一些系数）（这样看，residual physics难道都是在让深度学习模型来补上理想物理模型和现实情况之间的差距吗，也就是把因理想化假设导致的损失给弥补掉？毕竟物理规律也只是我们人抽象出来的，一定情形下的一种共性），一个是physical fusion（这个的大概思想是，把物理预测值当成是另一种模态的补充信息，让模型把纯数据驱动的输出和基于物理规律的输出进行一个融合，得到最终的输出。这个的话，其实也是一股子深度学习的味道，毕竟我们并没有显式告诉模型怎么利用物理规律，只是把物理规律潜移默化的藏在了物理预测值里，要让模型自己去学。这样或许会类似我之前做可控图生成的时候的那个方法一样，不显式的控制，可能力度不足？不过好处就是避免了前面residual physics的坏处，即我们此时不需要物理预测值有很好的准确性，利用物理规律的时候也更加方便了）

3.损失函数角度

从损失函数角度把物理规律和深度学习结合，就是让物理规律作为网络的一个正则化项，约束它不能偏离物理规律过远

Future outlook and conclusions

DART

Posted on 2025-05-09 In 论文阅读 , cv , prompt learning

问题设定：

要解决的是利用CLIP-based model用于下游的图像分类的问题

创新点：

提出学习两种模态的prompt（即text和image），而之前的工作大多是只学习text-modal prompt

学习的prompt之间是有联系的，即用不同的test sample学习prompt时，由前一个test sample习得的prompt会部分地继承到下一个prompt里。继承的程度则由前一个prompt的效果决定，效果越好、继承的越多

Method

1.Dual-Modal Online Prompting in DART

它主要就是对textual modal的prompt进行改进。之前的工作，一般都是一个prompt套到多个类上，比如this is a photo of bird/dog/...... 而论文希望改进，它给每个类别都设计了自己的prompt，并且后续可以学习

另外，之前的工作大部分只是在调整textual modal的prompt，但该论文希望visual modal也能有prompt，于是也提出了一个prompt。但这里就没有像上面那样给每个类别都设计一个prompt，而是只设计了一个prompt

这里的两种prompt都是instance-level的，就是每个样本的都不一样

2.Adaptive Knowledge Retention in DART

之所以它希望能做到knowledge retention，是因为测试的时候数据本就不多，并且它是online learning

knowledge retention的方法就是引入一组prompt，包括visual和textual的。它们记录了历史信息，有点类似于LSTM里的隐状态。具体来说，它引入的这组visual和textual prompt，都是分类别的。这里一定要和上面的prompt区分开，二者不是同一个东西。这里引入的我们可以叫它历史prompt，上面的是instance-level的。然后历史prompt的学习是依赖于instance level的，有点Moco里的动量更新的感觉；而instance level prompt的初始化又是依赖于历史prompt的

这里的核心思想，或者说假设，是：对一个输入样本，它属于哪个类的概率更高（这个概率是用了visual和textual prompt的），我们认为那个类对应的instance-level visual/textual prompt质量越高，也会越多地把它融合进历史prompt里

3.The Optimization of DART

上面要用到的instance-level visual/textual prompt，其实应该是用历史prompt初始化后，需要学习的

它具体的学习过程如下：对一幅输入图像，先做B次augmentation，得到B幅图像（B即batch size），然后利用初始化的prompt计算出各个图像属于各个类别的概率分布，再筛选出比较自信的结果，取平均得到原始样本属于各个类别的概率，最后我们优化原始样本的那个概率分布，希望它的信息熵尽可能小，也就是要尽可能确定，逼近一个独热向量（优化应该是梯度下降吧，还没看代码）

FineFMPL

Posted on 2025-05-09 In 论文阅读 , cv , Incremental Learning

背景知识：

它研究的是Few-Shot Class Incremental Learning (FSCIL)，即少样本增量学习。简单来说，它研究的是增量学习，但新类别的数据很少的情况。它一般会遇到两个问题，一是灾难性遗忘，二是对新类别的样本过拟合（因为新类别的样本很少）

method：

大体上，本论文提出利用VLM（Vision-Language Model），去提取图像的粗粒度的全局信息和细粒度的细节信息，然后再结合文本信息（这块再细看，没懂），最后把所有提取出的信息结合起来进行分类

0.背景

先明确，论文主要是利用VLM，因而模型有两个模态的encoder，即image模态和text模态的encoder

1.visual probe prompt

它是用来提取出图像中的discriminative information的，也就是提取出关键信息。具体来说，我们希望用它提取出一幅图像的global-level feature和object-level feature（前者粗粒度，后者细粒度）

大体操作：对于一幅图像，我们会把它处理成多个patch组成的seq（因为这里VLM的visual encoder是ViT），然后在最后面加入一个可学习的VPP token（即visual probe prompt token）（当然，前面的CLS token也还是在的，后面要用），组成新的序列，然后送到encoder里。

（1）提取粗粒度信息，可以把这个seq送到encoder里，让它照常输出，并且还是把CLS token的内容当作全局信息，也就是我们认为的global-level feature。

（2）提取细粒度信息，则不能直接把seq送到encoder里，而是要走一个新的分支。它也是ViT，但到了后面会有一点不同的操作。具体来说，到了transformer的倒数第二层后，它会取出VPP token在这一层的结果，然后用它对该层的所有token的embedding进行一个scale（也就是进行哈达玛积），并通过一个MLP以及残差连接，然后才送到transformer的最后一层。得到最后一层的输出后，会用VPP token的embedding计算各个patch对应的token的权重，细粒度的object-level feature就是CLS token最后的输出再加上各个patch token的加权求和。重点在于这个权重，它衡量了各个patch的重要性，它认为重要的patch，最后权重比较大，加的比较多，也就对应了论文希望的object-level feature可以捕捉细粒度信息的目的

2.Visual Prototypes Constructions

它是用来防止灾难性遗忘的。具体来说，它会提取出已经学过的各个类别的global-level feature和object-level feature，也就是所谓global-level prototype和object-level prototype，用来保存学过的知识

关于如何获得这两种prototype：

（1）获得global-level prototype：正常把图片输入到ViT的encoder后，将输出的CLS token的特征视作该图像的全局信息。然后，如果要获得第i个类的global prototype，就把所有的第i类的图像的输出CLS token取个平均。其它类的global-level prototype类似

（2）获得object-level prototype：ViT中间各个层会有注意力矩阵，若有H个头，则每层有H个注意力矩阵。把所有层的第i个头的注意力矩阵累乘（逐元素），然后取出CLS与其它token的关系性的那一行（其实也就是第一行吧，后面可以看代码验证一下）。我们可以认为这就是第i个头综合所有层的信息提取出来的，各个token的重要性，它也就是一个长度为N的一维向量（如果seq length为N+1，除掉CLS token后则有N个token）。类似地，我们算出所有头对应的token的重要性向量，把它们累加起来，就得到最终的重要性向量。用它对patch seq进行加权求和（其实也就是一个向量点乘了），得到object-level prototype。之所以是object-level，主要还是因为它是由各个patch token加权求和得到，而权重又是反映了各个patch的重要性。对于一幅图像可以如此得到一个object-level prototype，类似地，对所有图像都可以得到它们自己的object-level prototype。因为我们要分类，所以还是像上面那样，第i个类的所有图像的object-level prototype求个平均，得到第i个类的object-level prototype。其它类做法类似

3.Textual Context Prompt

这也算是文章的主要创新点了，因为它就是在强调，它提取了两种模态的信息，然后要用这两种模态的信息共同对图像进行分类

它是由上面的global-level prototype和object-level prototype计算得到的。大体上是先将两个level的prototype经过一个MLP，以实现信息融合，然后再用融合后的信息去分别和两个prototype进行哈达玛积，再经过sigmoid函数，得到一个类似注意力的权重，最后用它计算得到textual encoder的输入

细节上看起来有点复杂，但流程上也就是把两个visual prototype进行一个融合和重点提取，然后送到textual encoder，让它输出我们期望的global-level textual prototype和object-level textual prototype

之所以能用visual prototype得到textual prototype，主要也是像论文里说的那样，VL model有cross-modal knowledge和能力

4.Inference

最后推理的大概流程就是利用图像的global feature和object feature与我们提前计算好的visual和textual prototype进行匹配，得到分类的概率

具体来说，先是用visual prototype进行匹配。先将图像global/object feature经过一个MLP，同时visual prototype也通过一个MLP，然后将二者的结果进行点积，计算余弦相似度。比如，图像的global feature会和各个类别的global-level prototype进行点积，得到从全局角度上看，图像属于各个类别的logit。类似的可以得到从细节角度上，图像属于各个类别的logit。二者相加即得到最终的，从图像角度来看，输入图像属于各个类的logit（当然，论文里说了，二者相加不是直接相加，object-level的logit有个权重。这个就纯粹是小trick了）

至于textual prototype的匹配，和上面的过程基本一致，只是少了经过MLP的一步，直接进行相似度计算，也可以得到一个从文本角度来看，输入图像属于各个类的logit

最终，两个模态的logit相加，得到最后的，输入图像属于各个类别的logit。这也就是论文里说的，用两个模态的信息进行相互辅助的分类（同样的，这里也有一个权重，是图像角度的logit。同样是trick了）

疑问：

object-level特征为什么要用VPP的特征做scale？目的在哪里？各个patch的权重的计算意义何在？为什么认为patch的feature越接近VPP feature，这个patch越重要？VPP的意义到底是？

论文里最后进行推理的时候，visual那个分支上，反而feature和prototype都要经过一个MLP，说是为了减小数据之间的gap；但textual那个分支上却并没有经过MLP。可是feature是通过image得到的，和visual prototype之间的gap应该没有和textual prototype之间的gap大吧？MLP为什么不放到textual那个分支上？

实现Trie（前缀树）

Posted on 2025-05-08 In algo , 算法面试题

原题链接在这

题目描述：请你实现 Trie 类：

- Trie() 初始化前缀树对象。
- void insert(String word) 向前缀树中插入字符串 word 。
- boolean search(String word) 如果字符串 word 在前缀树中，返回 true（即，在检索之前已经插入）；否则，返回 false 。
- boolean startsWith(String prefix) 如果之前已经插入的字符串 word 的前缀之一为 prefix ，返回 true ；否则，返回 false 。

示例：

输入
["Trie", "insert", "search", "search", "startsWith", "insert", "search"]
[[], ["apple"], ["apple"], ["app"], ["app"], ["app"], ["app"]]

输出
[null, null, true, false, true, null, true]

解释
Trie trie = new Trie();
trie.insert("apple");
trie.search("apple"); // 返回 True
trie.search("app"); // 返回 False
trie.startsWith("app"); // 返回 True
trie.insert("app");
trie.search("app"); // 返回 True

提示：
- 1 <= word.length, prefix.length <= 2000
- word 和 prefix 仅由小写英文字母组成
- insert、search 和 startsWith 调用次数 总计 不超过 3 * 104 次

思路：

这题明确说了是考Trie树，事实上也确实如此，没有别的技巧手法。所以这里就讲一下Trie树是什么，它的原理又是什么

灵茶山艾府的题解写的很好。他的引入也比较好，假设一个字符串只有两种字符，我们可以怎么表示这个字符串？不妨记其中一种字符为a，另一种字符为b，然后，比如字符串是abbaabb，从左到右，我们可以把它看成是一颗二叉树，a就是左儿子，b就是右儿子，这样，这个字符串可以被一个二叉树所唯一地表示。然后，这一题里主要有三个要求（初始化暂且先不管），第一个是insert，也就是向前缀树中插入字符串，那么在这里只有两种字符的例子中，就还是构造相应的二叉树即可；
第二个是search，也就是从根节点出发，搜索一下，有没有待搜索的字符串的对应的二叉树。考虑到有可能一些字符是另一些字符的前缀，例如，abab和ababaa，所以我们考虑给每个节点再附带上一个end属性，用来表征它是不是一个终止节点。从根节点出发，走到一个终止节点，就代表遍历了一个字符串。这样，search的时候，我们从根节点出发，一个一个搜索字符串里的字符，不仅要全部都出现，而且在搜索到了最后一个字符时，对应的节点必须end属性为true
第三个是startsWith，它要判断给定的一个字符串是否为已经出现过的某个字符串的前缀。与第二个要求对比，其实就是，搜索的时候，我们只要求所有的字符一定要出现，且不要求搜到最后一个字符的时候，end属性一定为true，此时我们不关注end属性了，为true或为false都行

那么，现在落实到题目里的要求，会出现所有的小写英文字母，那也就是26种字符，则我们构造的树就变成了26叉树了。至于insert，search以及startsWith三个操作，原理还是没变的

代码：

感觉主要是很久没用struct了，有点不熟练啊……
简单看一下三个函数的部分，insert，那就是把给定要插入的字符串从左到右遍历一下，每个字符确保都插入；以及记得最后给end属性赋值为true
search部分，中间一旦搜索到某个字符不存在，则直接返回false；搜到最后一个字符了，再检查一下end是true还是false
startsWith部分，类似search部分，只是不检查end了。所以其实可以把startsWith以及search的公共逻辑部分抽象成一个函数，那样看起来会简洁一点
